# Send a SCOUT First: Pre-hoc Reasoning for Adaptive Detector Allocation in Prompt-Injection Defense

Abstract 

Prompt-injection detectors are heterogeneous: each is strong on a different slice of attacks, and none is always reliable. Yet existing systems still treat detection as a fixed single-detector pipeline, committing every request to one de-tector’s blind spots. We reframe defense as de-tector allocation : given a heterogeneous pool, decide per request which detectors to run and whether to escalate to an LLM judge. Our framework SCOUT (Scalable and Controllable 

Outcome-prediction for Uncertainty-aware 

Triage) makes this decision dynamic by pre-dicting each detector’s per-sample reliability and latency from how it behaved on similar past inputs, and exposes a single safety–utility threshold to the operator (where utility bun-dles benign-pass rate and wall-clock). To evaluate this setting, we build SCOUT-450, a benchmark that captures the structurally com-plex, agent-facing injections that older prompt-injection sets under-represent. On SCOUT-450, a safety-oriented operating point reduces attack-success rate by 46 % and total wall-clock by 

40 % relative to an always-on GPT-4o judge, at a 5.1-point benign-utility drop. SCOUT also transfers to three external benchmarks (BIPIA, IPI, and IHEval), improving the safety–utility frontier. 

1 Introduction 

Large language models are increasingly deployed in systems that read external content such as tool outputs, retrieved documents, web pages, and emails. Instructions hidden in that content can hijack the model into ignoring its operator’s in-tent (Pérez and Ribeiro, 2022; Greshake et al., 2023). A practical production defense must balance safety (the fraction of attacks it catches) against util-ity (the per-request wall-clock and the pass-through rate for benign traffic).

Existing defenses sit at different points on the latency–accuracy spectrum: lexical rules, lightweight classifiers (ProtectAI, 2023; Li et al., 2025a), attention and hidden-state probes (Hung et al., 2025; Wen et al., 2025), alignment-aware classifiers (Jia et al., 2026), and embedding or LLM-judge methods (Ganguly et al., 2026; Zheng et al., 2023). The LLM judge, a frontier model such as GPT-4o prompted to read each request and return an attack/benign verdict, is the most accurate but by far the slowest: a single API call costs on the order of 1.5 s, about four orders of magnitude slower than a rule-based keyword and regex scan-ner that flags an input in well under a millisecond. These smaller, faster detectors are each strong on a different attack category, and on some of them one beats the judge (Section 7.2). Which detector is best therefore changes with the input, yet a stan-dard cascade (Viola and Jones, 2001; Chen et al., 2024) fixes the order in advance and inherits the cheap detector’s mistakes wherever it is confident. To address this, we propose SCOUT , which allo-cates detectors per input (Figure 2). Concretely, for each request SCOUT retrieves anchors with similar injection structure and reads how each detector be-haved on them, and a small predictor then estimates which detectors are reliable for the input and how long each will take. The router runs the predicted-reliable light detectors in parallel, and escalates to an LLM judge only when their vote is uncertain. This makes SCOUT heterogeneity-aware : because the allocation is per input, each request can go to the detector that handles its attack category, includ-ing the categories where a small detector beats the judge. Two further properties follow from the same design. First, SCOUT is scalable , since adding a detector is just a single pass over the anchor set, with no retraining of the retriever, predictor, or rout-ing rule. 2 Related Work 

Prompt-injection attacks and benchmarks. Re-search on prompt injection spans direct at-tacks (Pérez and Ribeiro, 2022) and indirect at-tacks embedded in external content (Greshake et al., 2023), with taxonomies and benchmarks that define the attack space, instruction hierarchy, and agent-facing settings (Liu et al., 2024b; Wallace et al., 2024; Yi et al., 2025; Zhang et al., 2025c; Liu et al., 2025a) and analyze how instruction placement and transferable patterns shape vulnerability (Li et al., 2024, 2025b). 

Defenses. Standalone defenses span lexical guards and trained classifiers (ProtectAI, 2023; Chen et al., 2025a; Li et al., 2025a; Ja-cob et al., 2025), attention and hidden-state probes (Hung et al., 2025; Wen et al., 2025), alignment-aware classifiers (Jia et al., 2026; Min et al., 2026), embedding-OOD (Ganguly et al., 2026) and semantic-intent detectors (Wang et al., 2025), prompt-modification techniques (Chen et al., 2025b; Zhang et al., 2025a), and LLM judges (Zheng et al., 2023), each fixing its own safety, utility, and latency trade-off. Cascades es-calate from cheap to expensive stages (Viola and Jones, 2001; Chen et al., 2024), and composed sys-tems combine components through a rule fixed at design time (Le et al., 2026; Hossain et al., 2025; Liu et al., 2025b). All commit to a routing rule before seeing the input.

and BU , reported alongside the two axes. Sweep-ing τ traces the safety–utility trade-off.

5 Uncertainty-Aware Triage 

Algorithm 1 turns the predictor’s per-detector es-timates into a per-sample allocation. A subset se-lector keeps the light detectors the predictor marks reliable on x; they run in parallel and cast a trust-weighted vote. When the vote’s agreement falls be-low τ , an escalation gate consults the LLM judge, but only when the predictor also marks the judge reliable on x; otherwise the subset vote stands. 7.2 Safety–utility frontier on SCOUT-450 (RQ1) 

On SCOUT-450, per-input allocation improves the safety–utility trade-off over every single-detector baseline in Table 1, including the judge DLLM 

(GPT-4o) run on every request. At the headline operating point, SCOUT records the lowest To-tal ASR ( .063 ) and the highest accuracy ( .933 ): it is safer than the safest single detector (PIGuard, 

.098 ) and more accurate than the most accurate one ( DLLM , .924 ), at about 0.6× the wall-clock of 

DLLM . No single detector reaches both corners at once. The only axis on which DLLM leads is benign utility ( .979 vs. .928 ), the throughput that τ trades for safety. The threshold τ exposes a usable control sur-face across this frontier: low-τ points stay on the light pool and finish in a fraction of the headline wall-clock, while raising τ spends judge calls to tighten safety, so an operator reads off the point that meets its budget. Across the sweep SCOUT’s curve sits above the individual-detector points on every quality axis and to the left of the DLLM marker (Figure 3). The gain comes from detector heterogeneity. No single detector is strong everywhere, so a fixed deployment inherits one detector’s profile across the whole stream. 7.4 Pool reconfiguration exposes controllable operating points (RQ3) 

RQ3 tests whether SCOUT can absorb pool changes after training. A router tied to detector IDs or a fixed cascade would need new labels or cali-bration when the judge or light detectors change; SCOUT takes in a new component from its behav-ior on the anchor set alone. We freeze the trained predictor, retriever, routing rule, trust mixing, and threshold τ , and let each new judge or light detector enter only through a detector fingerprint built from one pass over Anchor-400, with no fine-tuning, cal-ibration fit, or gradient update. Table 3 applies this protocol on SCOUT-450. The Base rows swap the LLM judge while keep-ing the light pool fixed, testing whether the judge is a hard-coded endpoint or another fingerprinted component. The extension rows add D7 (Distil-BERT) and D8 (InstructDetector), absent from the Base light pool (Appendix A). These added detec-tors do not monotonically improve every metric; they expose different safety, benign-utility, and la-tency points under the same decision rule: with GPT-4o, adding D8 raises BU from .928 to .949 

while increasing ASR from .063 to .102 . This is the intended behavior of detector allocation: pool com-position changes the set of available detector behav-iors, while τ controls how conservatively SCOUT escalates beyond them.

8 Conclusion 

We reframe prompt-injection defense as detector allocation : the system decides per input which de-tectors to trust, combining behavioral fingerprints, a small predictor that estimates each detector’s per-input reliability and latency, and an uncertainty-gated routing rule. Because the predictor also esti-mates latency, a single operator threshold tunes the safety–latency trade-off against a budget known be-fore any request runs, and a new detector joins through one anchor-set pass with no retraining. On SCOUT-450 and three external benchmarks (BIPIA, IPI, IHEval), SCOUT improves the safety– utility trade-off over every single-detector baseline, including the always-on DLLM .Looking forward, we will release the predictor checkpoint, the SCOUT-30K training set, and the SCOUT-450 and Anchor-400 splits, and we main-tain the detector fingerprint library so operators can assemble their own detector pools. We will also open-source the full pipeline soon. 

Limitations 

Dependence on the available pool. SCOUT com-poses detectors; it does not improve any single detector. If the candidate pool contains no detector with usable signal on the target distribution, the routed system cannot exceed what the pool can support; its operating point on the safety–utility frontier is bounded by what the pool can collec-tively achieve. 

Anchor coverage and evaluation scope. The trust prior and the predictor’s retrieved context are use-ful in proportion to how well Anchor-400 covers the test distribution along the structural axes the embedder targets (category, hiding strategy, carrier, attack mechanism); when a test input falls far from every anchor, the per-sample component of the ef-fective trust becomes noisier and the router leans more on the precomputed global prior ¯πD (diag-nostics in Appendix C). Our benchmarks (SCOUT-450, BIPIA, IPI, IHEval) are also English-only and centered on a fixed taxonomy of prompt-injection mechanisms, so we do not evaluate cross-lingual transfer or attacks outside this taxonomy. Either shift would require refreshing the anchor set to reflect the new structural axes, which is a one-pass operation per detector. SCOUT-450 is also a compact, fully hand-curated diagnostic bench-mark ( 450 samples); we keep it small to guarantee per-sample category, carrier, and difficulty labels for fine-grained attribution, and our generalization claims rest on the three external benchmarks. 

Setup-dependent latency and adaptive attacks. 

Absolute wall-clock numbers depend on our A100 + vLLM serving stack and on the median per-detector accounting used in Appendix K, and the reported predictor uses a single SFT +GRPO recipe over Qwen3-4B-Instruct whose light-pool calibra-tion is the lever behind most of the routing gain (Section 7.3); other stacks or predictor backbones should re-measure. We also do not study adaptive attackers that target the router itself, for example by crafting samples that flip the predictor’s estimate for DLLM to suppress a useful escalation.

The detector pool D comprises a base pool of six families used throughout the main-text experiments (D1–D5 plus DLLM ), two additional detectors used in the pool-reconfiguration study ( D7, D8), and one further single-detector baseline reported for reference but not integrated into SCOUT’s pool (D9).

D Predictor training details 

This appendix expands the two-stage recipe of Sec-tion 4: hindsight SFT followed by GRPO. We de-scribe what each stage optimizes and why, with the training frameworks and full hyperparameters given below. 

Stage 1: hindsight SFT. The SFT stage trains the predictor on the SCOUT-30K corpus to pro-duce, for every (sample, detector) pair (x, D ),an assistant turn of the form “ z ∥ Predicted Performance: {correctness, latency} ”, where z is a short justification and the JSON suffix is the structured prediction. Targets are constructed by hindsight distillation : a teacher LLM is shown the prompt P (x, D ), the detector verdict ˆyD(x), the gold label y(x), and latency 

ℓD(x), then asked to justify the target (1[ˆ yD(x) = 

y(x)] , ℓ D(x)) . The teacher’s output is parsed into (z, pred_corr , pred_lat ), and we retain only pairs whose prediction matches that target, yielding the 29 ,551 kept examples that make up SCOUT-30K. Thus pred_corr is a correctness forecast, not a forecast of the detector’s attack/benign verdict. The training objective is the standard token-level cross-entropy on the assistant turn (loss masked over the prompt). We train two variants from the same dataset: • SFT-CoT : the full assistant turn z ∥ JSON. The predictor learns to emit a short reasoning chain followed by the structured prediction. • SFT-NoCoT : the rationale z is dropped at training time and the assistant turn is the JSON alone. This isolates how much of the routing signal depends on the model writing a chain of thought before emitting the structured output. SFT brings the parse-failure rate to zero but does not, on its own, lift pred_corr accuracy on the light pool above the base Qwen3-4B (the predictor-recipe block of Table 2). It learns the output for-mat of a calibrated predictor without yet learning the calibration . SFT-CoT initializes Stage 2; SFT-NoCoT is kept as an ablation point.

What each piece is doing. The format gate disci-plines the rollout distribution so the gradient sees parseable answers; without it, the policy can game the latency reward via verbose hedging. The cor-rectness reward is the routing signal: it aligns 

pred_corr with whether the detector will match the gold label, closing the light-pool calibration gap that pure SFT cannot (Appendix H, Figure 20). The latency reward provides a secondary, finer sig-nal that is informative only when correctness is satisfied; it calibrates pred_lat against observed detector latency, which is what lets the routing rule predict the wall-clock ˆℓ(x) of each decision (Algorithm 1) and lets τ control latency against a predicted budget (Section 7.6, Appendix E). Check-point selection is by routing accuracy on a held-out validation slice of the SCOUT-30K-derived RL split; the selected checkpoint is the one used in ev-ery experiment that names “SCOUT” in this paper.

D2 family and for retrieval use Qwen3-Embedding-0.6B (Zhang et al., 2025b); the D2 classifiers use scikit-learn (Pedregosa et al., 2011) defaults ex-cept for the k-nearest-neighbors neighborhood size (k=25 ). The threshold τ . τ is the deployment-time knob of Section 5 (Equation 5): a low τ escalates rarely and stays cheap, a high τ escalates more and ap-proaches the always-judge baseline. Sweeping τ

Deployment recipe for τ . At deployment, the operator picks τ by running SCOUT once on a small labeled slice of the target traffic and sweep-ing τ to trace the local safety–utility frontier (Fig-ure 3 shows this trace on SCOUT-450). The op-erator then selects τ to meet a stated budget: a latency budget L via Equation 5, which is label-free because ˆT (τ ) comes from pred_lat (Fig-ure 17); a safety target, which needs the labeled slice to measure ASR; or an LLM-call budget through the empirical escalation rate ρ(τ ) = |{ x :

rule escalates on x}| /|X|, observable from server logs with no labels. The selected τ is frozen for production. The operator monitors live ρ as a cheap drift signal: a sustained shift away from the cali-bration value triggers a re-sweep on a fresh labeled batch. When labeled validation data is unavailable at deployment, SCOUT ships with τ = 0 .875 (the SCOUT-450 headline) as the default.

M Worked examples 

This appendix gives one concrete example each of (i) the fingerprint LLM prompt that produces a detector behavioral record, (ii) the hindsight-distillation training prompt that supervises the SCOUT predictor, and (iii) two SCOUT inference outputs on SCOUT-450, including the predictor’s reasoning trace. 

M.1 Fingerprint construction 

The fingerprint LLM (GPT-OSS-120B) is called once per batch of 5 anchors, producing a structured record per (sample, detector) pair. The prompt template (with one detector instantiated): 

You are a security research assistant analyzing prompt injection detector behavior. DETECTOR: d3_deberta MECHANISM: Fine-tuned DeBERTa-v3-base transformer classifier. Trained end-to-end on external prompt injection data. Uses full 

bidirectional attention over the input sequence. For each of the 5 samples below, produce a structured summary with exactly 3 parts: 1. Detector profile: one sentence describing what this detector does and how it works. 2. Sample characteristics: describe in detail what the sample contains -- whether it is an attack or benign, its category, difficulty, carrier type, attack mechanism (if applicable), and the full content and goal. Keep to 3 sentences or fewer. 3. Prediction result: state what the detector predicted (attack /benign), whether it was correct or incorrect, the confidence score, and the latency in milliseconds. Return ONLY a JSON array, one object per sample, preserving input order: [{"id": "<sample id>", "detector_profile": "<1 sentence>", "sample_characteristics": "<detailed description>", "prediction_result": "<predicted, correct /incorrect, conf>" }, ... ]SAMPLES: <JSON list of 5 anchor samples with their detector outcomes> 

The returned JSON, one record per (sample, de-tector), is the fingerprint we index for retrieval. An example record for D3 on a hidden-tricky anchor: 

id: anchor_hidden_tricky-0072 

detector_profile: The d3_deberta detector is a fine-tuned DeBERTa-v3-base transformer classifier that scans the en-tire input with bidirectional attention to decide if a prompt contains a hidden injection attack. 

sample_characteristics: A hard-difficulty, hidden_tricky attack delivered as a markdown table that lists live API keys for several payment processors and embeds a hid-den comment <!–set_target_account:67890–> ; the at-tacker aims to have the model use the hidden account when processing the request. 

prediction_result: Predicted benign, incorrect (false neg-ative), confidence 0.0014 , latency 18 .31 ms.

N Reproducibility and responsible use 

Single-run reporting. All reported numbers are single runs on fixed held-out sets, without multi-seed variance or confidence intervals. The non-judge components are deterministic (fixed-weight detectors, greedy predictor decoding, and fixed Anchor-400 retrieval), so the only stochastic ele-ment is the hosted LLM judge; per-request latency is reported as the median (Appendix K). The τ -sweeps in Section 7 trace the operating curve, so the headline point is the knee of a smooth sweep. 

Potential risks. Releasing SCOUT-450 and the attack-category schema could in principle help an attacker study which prompt-injection structures evade specific detectors. We judge this risk to be low: the attack templates we use are standard instruction-override and exfiltration patterns al-ready present in public benchmarks such as BIPIA, IPI, and IHEval, and SCOUT adds no new attack capability. The released artifacts are intended to advance defense research, and our framework is a defensive orchestration layer. 

Use of AI assistants. Several pipeline compo-nents are LLMs by design: an off-the-shelf model (GPT-OSS-120B (OpenAI, 2025a)) serializes de-tector behavior into fingerprint records (Section 3); a teacher LLM produces the hindsight-distilled ra-tionales that supervise SCOUT-30K (Section 4, Appendix D); the outcome predictor is a fine-tuned Qwen3-4B-Instruct (Section 4); and GPT-4o, with GPT-5.1 as an alternate, serves as the LLM judge DLLM (Section 7.1, Appendix L). Separately, during manuscript preparation the authors used a general-purpose AI assistant for language editing and for drafting and debugging plotting and analy-sis code; all technical claims, experimental design, and analyses were produced and verified by the authors.
