## 摘要
LLM和智能体可以从自然语言需求生成网络爬虫，但直接生成因依赖错误、选择器失效、模式不匹配以及异构页面结构而不可靠。我们提出一个受限、可验证的智能体框架，将LLM输出从自由格式代码转换为类型化的JSON收集器配置，结合六类型收集器分类法、模板和实用函数约束、静态Airflow DAG执行、基于规则的质量检查以及结构化反馈修正。在138个任务上的实验表明，该分类法支持基于描述的需求类型化，同时确认稳定实例化需要补充超过初始描述的来源、字段和执行约束。在80个独立来源验证的任务上，该框架以零执行阶段LLM令牌和最低平均挂钟时间运行，以适度的单次质量为代价，获得适合重复调度收集的可重用、确定性和可验证执行路径。这些结果使该框架成为重复开放网络数据收集的可重用、低成本且可验证的执行路径。

## Abstract
LLMs and agents can generate web scrapers from natural-language requirements, but direct generation remains unreliable because of dependency errors, broken selectors, schema mismatches, and heterogeneous page structures. We propose a constrained, verifiable agent framework that shifts LLM output from free-form code to typed JSON collector configurations, combining a six-type collector taxonomy, template and utility-function constraints, static Airflow DAG execution, rule-based quality checking, and structured feedback correction. Experiments on 138 tasks show that the taxonomy supports description-based requirement typing, while confirming that stable instantiation requires completing source, field, and execution constraints beyond the initial description. On 80 independently source-verified tasks, the framework runs with zero execution-stage LLM tokens and the lowest average wall-clock time, trading moderate one-shot quality for a reusable, deterministic, and verifiable execution path suited to repeated scheduled collection. These results position the framework as a reusable, low-cost, and verifiable execution path for repeated open-web data collection.

## 1. 核心问题与整体含义（研究动机和背景）
- **研究动机**：开放网络数据（新闻、政府公告、电商、学术出版物）的自动化收集需求日益增长，但传统管道依赖手动开发（需求分析、网页结构检查、爬虫编码、验证），周期长且复用性差。LLM 和智能体虽可从自然语言需求生成爬虫代码，但直接生成不可靠：网页结构异构性（结构脆弱）与 LLM 代码生成的随机不一致性（依赖错误、选择器失效、输出模式不匹配）之间存在核心冲突。
- **核心问题**：如何使开放式网络数据收集管道变得可验证，以便每个阶段的输入输出都能被检查、测量和审计，从而将不可控的失败转化为可控、可追溯的信号。
- **整体含义**：论文主张通过类型约束、模板约束、规则质量门控等工程手段，将 LLM 输出从自由格式代码转为类型化 JSON 配置，实现确定性的、可复用的、低成本的重复调度收集路径，而不是追求完美的智能体推理。

## 2. 提出的方法论
### 核心思想
- **设计哲学**：“受控的显式失败优于不受控的静默成功”——通过类型约束、Schema 验证和规则质量门控，使失败变成可审计、可追踪的信号。
- **三大支柱**：
    1. **类型化任务建模**：用六类型收集器分类法（Search、List、Detail、API、Interactive、File）对收集任务进行显式功能边界划分和组合规则定义。
    2. **模板和工具函数约束**：将智能体从自由代码生成转向配置生成——智能体只负责填充模板槽位（URL、选择器、参数、字段映射等），而模板硬编码了 Airflow 结构、工具函数骨架、错误处理和持久化逻辑。
    3. **小样本验证与质量反馈闭环**：在规模执行前通过小样本验证和质量检查拦截低质量配置，形成“生成→执行→检查→修正”闭环。

## 7. 优点
- **创新性**：用“类型化 JSON 配置 + 约束生成”范式替代“自由代码生成”，将 LLM 不可控性限制在配置生成环节，执行层完全确定。
- **可验证性**：每个阶段（需求理解、配置、验证、质量检查、反馈修正）都有显式约束和结构化输出，便于审计和回滚。
- **成本效率**：零执行阶段 LLM 令牌，低延时，适合高频重复采集；配置生成成本一次性摊销。
- **鲁棒设计**：采用规则质量检查（非 LLM 判断）、反馈约束（非自然语言反思）、失败黑名单单调剪枝，避免 LLM 自检失败问题。
- **工程实用性**：与 Airflow 调度兼容，支持验证/规模双阶段执行，提供质量门控和错误标签。
