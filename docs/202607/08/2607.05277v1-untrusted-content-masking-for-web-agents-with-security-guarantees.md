## 摘要
针对提示注入攻击提供安全保障的防御措施依赖于可信指令与不可信数据的严格隔离。在基于文本的环境（如工具使用API）中，这种分离自然产生：代理可以通过接口定义进行推理，而无需处理不可信内容。将这些保障措施扩展到Web代理面临一个根本性挑战：为了感知和交互，Web代理必须首先观察渲染后的页面，而页面中混合了可信与不可信内容。这种结构上的纠缠消除了安全保障所依赖的信任边界，从而削弱了针对Web代理的可证明防御。本文提出了一种简单有效的方法——不可信内容屏蔽（UCM），它在Web环境中恢复了这一边界。我们利用一个关键的结构性洞察：网页的文档对象模型（DOM）编码了足够的信息，无需读取内容即可区分可信和不可信区域。我们的框架通过以下方式实现：在不可信区域到达代理之前对其进行编辑，并通过具有严格权限分离的沙盒接口进行交互路由，从而使代理能够在观察和交互环境的同时与对抗性内容隔离。代码已公开提供。

## Abstract
Defenses that provide security guarantees against prompt injection attacks rely on strict isolation between trusted instructions and untrusted data. In text-based environments such as tool-use APIs, this separation arises naturally: agents can reason from interface definitions without ever processing untrusted content. Extending these guarantees to web agents faces a fundamental challenge: to perceive and interact with their environment, web agents must first observe the rendered page, which intermingles trusted content with untrusted content. This structural entanglement removes the trust boundary on which security guarantees depend, undermining provable defenses for web agents. In this paper, we present Untrusted Content Masking (UCM), a simple and effective approach that restores this boundary in web environments. We leverage a key structural insight: a webpage's Document Object Model (DOM) encodes sufficient information to distinguish trusted from untrusted regions without reading their content. Our framework exploits this by redacting untrusted regions before they reach the agent and routing interaction through a sandboxed interface with strict privilege separation, thereby enabling agents to observe and interact with their environment while remaining isolated from adversarial content. The code is publicly available.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：Web代理（如Claude Computer Use、OpenAI Operator）在浏览器中执行任务时，必须观察渲染后的网页。然而网页中可信内容（导航、官方列表）与不可信内容（用户评论、广告、第三方内容）混合在一起，导致传统的严格隔离防御（如文本工具API中的隔离）无法直接应用。这种结构上的纠缠使得代理面临严重的提示注入攻击风险——攻击者可以通过注入恶意文本操控代理行为。

- **研究动机**：现有防御分为启发式（无形式保障，可被绕过）和保证式（强制隔离但严重限制功能）。例如，单次规划提供控制流完整性但使代理对页面完全盲视；基于类型化API的规划需要大量站点基础设施。本文希望在不牺牲任务灵活性的前提下提供安全性保障。

- **核心挑战**：如何在不直接读取不可信内容的情况下，恢复可信与不可信之间的信任边界，使代理既能够观察和交互，又能避免被注入攻击。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：利用网页DOM结构天然编码的信息，在代理感知之前自动区分并掩码不可信区域，同时通过沙盒接口（分离模型）提供受限的访问途径。

- **关键技术细节**：
  1. **不可信内容屏蔽（UCM）**：
     - 基于DOM结构识别不可信区域（通过CSS选择器或手动标签）。
     - 在代理看到渲染页面前，将所有不可信元素替换为带有唯一标识符的标签（如“id: reviews:0”），保留布局和语义但隐藏实际内容。
     - 代理只能看到可信内容与占位符，完全避免接触对抗性文本。

  2. **隔离模型（Q-Model）**：
     - 当代理需要读取不可信内容时，不能直接访问，而是调用一个隔离的Q-Model（独立LLM实例）。
     - 代理指定目标元素ID、自然语言问题以及返回类型（仅限`bool`, `int`, `float`, `date`, `enum`），Q-Model读取实际内容并返回结构化响应。
     - 返回类型受限，使得注入指令无法作为自由文本传播给代理，从而确保控制流安全。
     - 对于需要自由文本的任务，提供用户审批的字符串输出回退（例如在WebArena实验中）。

  3. **自动边界推理（可选）**：
     - 对于被动诚实的页面所有者（不提供标签），使用LLM基于内容净化的DOM（只保留结构，去除所有用户可见文本）推断CSS选择器来标记不可信区域。
     - LLM迭代生成选择器，通过迭代检查遗漏来最大化覆盖率。

  4. **威胁模型假设**：
     - 攻击者可以完全控制不可信区域的内容（包括文本和图像），并假设一旦进入模型上下文即可任意控制输出。
     - 信任锚点：页面所有者诚实（不恶意添加内容）；防御依赖此假设。

## 7. 优点

- **简单有效**：利用现有DOM结构，无需复杂修改页面或改造代理架构。
- **可证明安全**：通过设计保证代理从不处理不可信内容，而非依靠启发式检测。
- **实用部署友好**：可与主动所有者标签（类似ARIA标签）结合，也可通过LLM自动推理适用于被动所有者。
- **保持灵活交互**：代理仍可使用标准ReAct循环，无需切换为盲规划模式。
- **成本可接受**：在合理范围内，且随模型进步有望降低。
- **系统兼容性**：与下游数据流防御（如CaMeL、FIDES）互补。

## 8. 不足与局限

- **信任假设依赖诚实页面所有者**：如果页面被完全恶意控制（例如钓鱼网站），UCM无法保护。但作者指出这属于根本性不安全，无论防御如何。
