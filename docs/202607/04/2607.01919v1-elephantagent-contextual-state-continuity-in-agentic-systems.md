## 摘要
智能体系统通过调用外部工具和保持持久性内存来增强其能力。然而，这些外部依赖引入了新的攻击面。最近的工具和内存投毒攻击表明，恶意构造的工具描述符和中毒内存可以隐蔽地偏置智能体行为。这些威胁反映了一个更深层次的问题：智能体用于规划和执行的上下文状态缺乏可验证的连续性。我们提出ElephantAgent，一种强制实现上下文状态连续性的协议，以防御上下文状态投毒。受先前状态连续性机制（例如Nimble）的启发，ElephantAgent将这种保护扩展到智能体系统不断演变的上下文状态。我们将上下文状态定义为智能体整个上下文中受限制的、安全关键的子集（例如，工具状态和内存）。在处理每个查询之前，ElephantAgent重新计算本地上下文状态的摘要，并与最新授权的摘要进行验证。通过使用复制的可信硬件，ElephantAgent维护一个线性化的授权上下文状态转换账本，并检测带外状态篡改。为了处理带内语义滥用，ElephantAgent额外提供历史可追溯性，从而实现有条件的事后审计和恢复到已知良好的先前状态。

## Abstract
Agentic systems enhance their capabilities by invoking external tools and maintaining persistent memory. However, these external dependencies introduce novel attack surfaces. Recent tool and memory poisoning attacks show that maliciously crafted tool descriptors and poisoned memory can covertly bias agent behavior. These threats reflect a deeper issue: the lack of verifiable continuity in the agent's contextual state for planning and execution. We present ElephantAgent, a protocol that enforces Contextual State Continuity to defend against contextual state poisoning. Inspired by prior state-continuity mechanisms (e.g., Nimble), ElephantAgent extends this protection to the evolving contextual state of agentic systems. We define the contextual state as the bounded, security-critical subset of the agent's entire context (e.g., tool state and memory). Before processing each query, ElephantAgent recomputes the digest of the local contextual state and verifies it against the latest authorized digest. Using replicated trusted hardware, ElephantAgent maintains a linearizable ledger of authorized contextual state transitions and detects out-of-band state tampering. To handle in-band semantic abuse, ElephantAgent additionally provides Historical Traceability, enabling conditional post-hoc audit and recovery to a known-good prior state.

## 1. 核心问题与整体含义（研究动机和背景）
- **目标**：智能体系统（Agentic Systems）通过调用外部工具和持久化记忆来扩展能力，但这也引入了新的攻击面。
- **威胁**：工具描述符中毒（如 MCP “rug pull” 攻击）和记忆中毒攻击可隐蔽地偏置智能体的规划和执行，使其成为混淆代理（confused deputy）。
- **根本原因**：当前智能体系统缺乏对上下文状态（工具状态 + 记忆）的可验证连续性。状态可以被带外添加、删除或修改而不被检测。
- **整体含义**：ElephantAgent 旨在通过强制上下文状态连续性来防御此类攻击，为智能体系统建立可验证的状态演进历史。

## 2. 方法论：核心思想、关键技术细节与流程

### 核心思想
- 将智能体的上下文状态（Contextual State）定义为其整个上下文中**有界、安全关键**的子集（例如工具描述符和三元组记忆：长期指令、查询级转录、托管工件）。
- 通过复制可信硬件（TEE）维护一个**线性化、仅追加的账本**，记录所有授权的上下文状态转换摘要在每次查询前，本地 Context Guard (CG) 重新计算摘要并与账本中的最新授权摘要比对，检测带外篡改。
- 额外提供**历史可追溯性**，支持有条件的事后审计和恢复到先前的已知良好状态，以应对带内语义滥用。

### 关键技术细节
- **组成**：
  - **Context Guard (CG)**：运行在 MCP Host 内的 TEE 中，负责本地状态验证和更新。
  - **State Continuity Module (SCM)**：部署于复制 TEE（如 Intel TDX）中的线性化账本服务，基于 Nimble 实现。
- **工作流程**（Algorithm 5）：
  1. **初始化**：通过 Approval Policy（如 TOFU）批准初始状态，计算摘要并提交至 SCM 建立账本。
  2. **状态验证**（Algorithm 3）：每次查询前CG从本地取工具和记忆，计算摘要，请求 SCM 的最新授权证明（含新鲜 nonce），验证签名和摘要匹配。
  3. **状态更新**（Algorithm 4）：执行计划后，先通过预览（PreviewApply）无副作用地计算下一状态摘要，提交至 SCM，获得签名收据后再持久化本地副作用（commit-before-finalize）。
- **安全性**：形式化证明了上下文状态连续性（Lemma 1, Corollary 1）和历史可追溯性（Lemma 2, Corollary 2），基于标准密码学假设（碰撞抗性哈希、签名不可伪造等）和 TEE 信任假设。

## 6. 主要结论与发现
- **初始化开销大但一次性**：ElephantAgent 平均初始化 1800 ms，是 NoGuardAgent（219 ms）的 8.22 倍，主要是远程证明和 SCM 设置，后续查询中摊销。
- **每查询开销极小**：
  - 状态验证：4.23–4.33 ms
  - 状态更新：5.25–5.39 ms
  - 数据加载额外时间 <1 ms
- **端到端总时间开销**：1.02x – 1.04x，表明 ElephantAgent 可集成到现有 Agent 系统而不引入明显延迟。
- **消融实验**：AttestedSrvAgent 增加了 1.51x 初始化时间和 1.12x–1.22x 工具调用时间，但仍较小。
- **安全有效性**：基于形式化证明，能检测带外工具/记忆篡改，支持事后审计和恢复。

## 7. 优点
- **基于验证的防御**：与现有基于检测或归因的方法不同，ElephantAgent 通过密码学验证直接阻断带外篡改，无需依赖启发式。
- **形式化安全属性**：清晰定义了上下文状态连续性和历史可追溯性，并给出完整证明。
- **通用性好**：协议不限于 MCP，可应用于 CLI 等其他 agent 框架（附录 A）。
- **实现实用**：基于成熟的 Intel TDX 和 Nimble 账本，无需芯片级修改，性能开销低。
- **支持恢复**：历史可追溯性提供了针对带内语义滥用的第二道防线，允许审计和回滚。

## 8. 不足与局限
- **LLM 通信安全不在范围**：未保护 Agent 与外部 LLM 服务之间的信道，若该信道被攻破，仍可操纵提示/响应。
- **可用性风险**：攻击者可删除本地账本或快照，导致服务中断（fail-closed），且无法进行事后审计/恢复。TEE 密封不保证可用性。
- **恢复范围有限**：只能恢复限定的上下文状态（工具+记忆），不能恢复完整的运行时上下文。
- **初始信任根**：原型中 Approval Policy 使用 TOFU，可能被锚定恶意状态；建议生产环境使用签名验证。
- **未防御高级语义滥用**：即使状态连续性通过，带内注入（如 prompt injection）仍可造成危害，需依赖历史可追溯性且要求快照可用。
- **未覆盖所有攻击面**：如 TEE 本身的侧信道攻击、SCM 多数节点失效导致锁死等属于范围外。
