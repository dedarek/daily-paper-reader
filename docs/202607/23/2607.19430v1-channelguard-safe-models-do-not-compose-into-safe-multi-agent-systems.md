## 摘要
多智能体大语言模型（LLM）应用将规划器、工作智能体、验证器和合成器链接起来，而智能体之间的每一跳都是一个不受监控的通道，攻击者可以通过该通道走私指令。现有的防御措施仅保护输入边界（IBProtector、Llama Guard、困惑度过滤器、SmoothLLM），或者在应用外部作为不透明的、随机的提供商端过滤器运行。我们表明，这一差距带来了一个极少被衡量的后果：在涵盖8个攻击家族、5种防御措施和3个模型后端的2,100条轨迹评估中，一个在标准报告下看似完全安全的无防御流水线（在工具和内存投毒上攻击成功率为0.000），其安全性几乎完全归功于云提供商的服务器端过滤器（在Azure GPT-5上60次阻断中有54次），而在没有此类过滤器的后端上，则悄然转变为依赖智能体模型自身的对齐。仅报告结果的评估掩盖了这种依赖性。我们提出了ChannelGuard，这是一个免训练的深度防御框架，在每条智能体间通道上放置信息瓶颈门；每个门通过嵌入相似度将通道文本与对抗性短语库进行评分比对，并确定性地执行通过、压缩或阻断操作，且不增加任何LLM调用，同时一种归因方法记录了是哪一层阻止了每次攻击。ChannelGuard的工具输出门在应用层阻断了30次工具投毒攻击中的30次，在Azure GPT-5、Anthropic Sonnet 4.5和Anthropic Haiku 4.5上表现完全一致，而无防御流水线在不同后端间则发生彻底偏移；它还将提示注入攻击的成功率降低了一半（从0.333降至0.167），并完全保留了GSM8K的准确率（0.867）。白盒自适应复述可以逃避每个嵌入门，而扰动与投票基线则表现更好。扩展附录增加了基线、消融实验、参数扫描、良性保留分析以及评判审计（kappa = 0.900），总成本为47.36美元。

## Abstract
Multi-agent LLM applications chain a planner, worker agents, a verifier, and a synthesizer, and every hop between agents is an unmonitored channel through which an adversary can smuggle instructions. Existing defenses guard only the input boundary (IBProtector, Llama Guard, perplexity filters, SmoothLLM) or run outside the application as opaque, stochastic provider-side filters. We show this gap carries a consequence rarely measured: on a 2,100-trace evaluation across eight attack families, five defenses, and three model backends, an undefended pipeline that appears fully safe under standard reporting (attack success 0.000 on tool- and memory-poisoning) owes that safety almost entirely to the cloud provider's server-side filter (54 of 60 blocks on Azure GPT-5), and silently shifts to the agent model's own alignment on a backend without such a filter. Outcome-only reporting hides this dependence. We present ChannelGuard, a training-free defense-in-depth framework placing information-bottleneck gates on every inter-agent channel; each scores channel text against an adversarial phrase bank by embedding similarity and deterministically passes, compresses, or blocks it, adding no LLM call, while an attribution method records which layer stopped each attack. ChannelGuard's tool-output gate blocks Tool Poisoning 30 of 30 at the application layer, identically across Azure GPT-5, Anthropic Sonnet 4.5, and Anthropic Haiku 4.5, whereas the undefended pipeline shifts entirely across backends; it also lowers Prompt Injection attack success by half (0.333 to 0.167) and preserves GSM8K accuracy exactly (0.867). White-box adaptive paraphrase evades every embedding gate, where a perturb-and-vote baseline does better. An extended appendix adds baselines, ablations, sweeps, a benign-preservation analysis, and a judge audit (kappa = 0.900), at a total cost of 47.36 USD.

## 1. 核心问题与整体含义
*   **研究背景**：现代多智能体 LLM 应用通常由规划器、工作智能体、验证器和合成器串联而成。然而，智能体之间的通信信道（如工具输出、共享内存读取）缺乏监控，成为攻击者走私恶意指令的“盲区”。
*   **核心问题**：现有的防御手段（如 IBProtector、Llama Guard）仅关注输入边界，或依赖云服务商不透明的服务器端过滤器。这导致了一个被忽视的现象——**“机制替代”**：一个看似安全（攻击成功率为 0）的系统，其安全性可能完全依赖于云端的黑盒过滤，而非应用自身的防御；一旦更换后端模型，安全机制可能完全失效。
*   **整体含义**：安全的模型无法简单组合成安全的多智能体系统。仅报告结果的评估指标（ASR）掩盖了系统对云端过滤器的隐性依赖，开发者无法据此判断系统的真实鲁棒性。

## 2. 提出的方法论
*   **核心思想**：提出 **ChannelGuard**，一种免训练的深度防御框架。该框架在多智能体流水线的每一条信道上设置“信息瓶颈（IB）门”，通过压缩信息流来阻断攻击，实现应用层可控、确定性的防御。
*   **关键技术细节**：
    *   **六道 IB 门**：覆盖了用户输入、规划器到工作智能体、工具输出、内存读取、工作智能体输出以及合成器输入六个关键信道。
    *   **门控机制**：基于句子嵌入相似度进行评分。计算信道文本与预定义的对抗性短语库之间的最大余弦相似度。
    *   **决策逻辑**：根据风险评分执行三态决策：
        *   **阻断**：风险评分高于阈值 $\theta$，丢弃文本。
        *   **压缩**：风险评分适中，保留前 $N$ 个句子（假设恶意指令常位于末尾）。
        *   **通过**：风险评分低于阈值 $\alpha\theta$，原样放行。
    *   **辅助组件**：包含一个不确定性感知的验证器（5次蒙特卡洛采样）和一个三重过滤器，用于最终输出的安全把关。
*   **算法流程**：系统在运行时，每个信道文本都会经过门控算法。一旦某道门判定为“阻断”，流水线即刻短路返回拒绝响应，不再消耗后续的 LLM 调用成本。

*   **数据集与场景**：
    *   使用 **HotpotQA**（30个样本）作为主要评估任务，模拟多跳问答场景。

*   **Benchmark（攻击家族）**：涵盖 8 种攻击类型，包括提示注入、工具投毒、内存投毒、慢漂移内存、共识升级内存、角色混淆内存以及白盒自适应复述。
*   **对比方法**：
    *   **基线系统**：无防御流水线。
    *   **现有防御**：IBProtector（输入边界保护）、LLM-Guard（分类器）、Perplexity-Filter（困惑度过滤）、SmoothLLM（扰动投票）。
*   **模型后端**：主要使用 Azure GPT-5，并在 Anthropic Sonnet 4.5 和 Haiku 4.5 上进行了跨后端验证。

## 4. 资源与算力
*   **算力环境**：实验在单台工作站上运行，使用 8 线程并行。嵌入模型在本地 CPU 上运行，无需 GPU 加速。
*   **成本统计**：总计消耗 30,769 次 LLM 调用，API 成本为 **47.36 美元**。其中 Azure GPT-5 和 Claude Sonnet 4.5 为主要消耗来源。
*   **效率表现**：ChannelGuard 在提示注入攻击上比无防御基线快 3.30 倍（因早期阻断节省了后续调用），整体平均快 1.19 倍。

## 5. 实验数量与充分性
*   **实验规模**：共评估了 **2,100 条轨迹**，并进行了 100 组跨家族评判审计。
*   **实验设置**：
    *   **消融实验**：对六道门逐一移除进行测试，验证各门的贡献。
    *   **参数扫描**：对阈值 $\beta$ 和短语库大小进行了超参数敏感性分析。
    *   **归因分析**：提出了“逐轨迹归因”方法，精确记录每次阻断是由哪一层（IB门、提供商过滤器、验证器等）完成的。
*   **充分性与公平性**：实验设计非常详尽，不仅对比了攻击成功率（ASR），还引入了良性保留率（BPR）以防止过度拒绝。通过固定样本 ID 的配对比较，确保了对比的公平性。归因分析揭示了“机制替代”现象，弥补了传统评估的盲点。

## 6. 主要结论与发现
*   **防御效果**：ChannelGuard 在应用层 **100% 阻断了工具投毒攻击**，且在不同模型后端间表现一致；将提示注入成功率降低了 50%（从 0.333 降至 0.167）。
*   **机制替代现象**：无防御流水线在 Azure GPT-5 上的零攻击成功率，有 90% 归功于 Azure 的服务器端过滤器；而在 Anthropic 后端上，安全机制完全转变为依赖模型自身的对齐。ChannelGuard 提供了应用层可控的阻断，消除了这种不确定性。
*   **性能保真**：在 GSM8K 上保持了 0.867 的准确率，未损害模型的推理能力。
*   **局限性**：白盒自适应复述攻击可以通过语义等价但词汇差异大的文本绕过嵌入门；此时 SmoothLLM 基线表现更好。

## 7. 优点
*   **创新性视角**：首次深入揭示了多智能体系统中“安全模型组合不等于安全系统”的问题，并指出了现有评估指标 ASR 的欺骗性。
*   **轻量高效**：无需训练，不增加额外的 LLM 调用，利用本地嵌入模型即可运行，且具有短路机制，能降低系统延迟。
*   **可解释性强**：通过归因分析，能明确告知开发者是哪一道门拦截了攻击，而非仅给出一个黑盒结果。
*   **跨后端一致性**：防御效果不依赖于特定云厂商的过滤器，解决了“机制替代”带来的部署风险。

## 8. 不足与局限
*   **对抗绕过风险**：对于拥有短语库知识的白盒攻击者，自适应复述攻击能有效绕过防御，说明静态短语库的泛化能力有限。
*   **压缩策略缺陷**：压缩策略假设恶意指令位于文本末尾，若攻击者将指令置于前两句，IB-0 门存在约 23.4% 的泄漏率。
*   **样本规模限制**：主要评估基于 30 个样本的配对测试，虽然进行了显著性检验，但在更广泛场景下的统计显著性仍需更大样本验证。
*   **语言限制**：当前的短语库和嵌入模型主要针对英文，未讨论多语言场景下的防御效果。
