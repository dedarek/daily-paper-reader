## 摘要
使用工具的大语言模型智能体越来越多地读取不可信内容，同时持有支付、邮件、客户关系管理和基础设施API等具有副作用工具，然而常见的框架默认设置仍将工具暴露与授权混为一谈。我们审计了LangChain/LangGraph、LlamaIndex和Stripe Agent Toolkit是否在执行前对每个模型发出的调用（带具体参数值）重新授权。在固定的公开源码提交中，三者默认提供能力门，但默认均未提供确定性的、默认拒绝的按调用值授权门。我们提出ScopeGate，一种针对智能体工具调用的五阶段策略决策点/策略执行点：范围、授权、金额上限、幂等性和默认拒绝。评估显示，相同的未经授权的支付调用在LangChain的默认调度下执行（附带LlamaIndex概念验证），但被ScopeGate拒绝；受测控制报告0/48静态绕过、0/29未经授权尝试（40次迭代自适应运行）、0/10良性错误拒绝，以及Latam-GPT支付智能体在10/10次中受到遏制。ASR表示尝试的未经授权操作，遏制并非万能，部署层声明基于对实测模型类别的推断，且未声称任何CVE。

## Abstract
Tool-using LLM agents increasingly read untrusted content while holding side-effecting tools such as payments, email, CRM, and infrastructure APIs, yet common framework defaults still conflate tool exposure with authorization. We audit whether LangChain/LangGraph, LlamaIndex, and the Stripe Agent Toolkit re-authorize each model-emitted call, with concrete argument values, before execution. Across pinned public-source commits, all three provide capability gating by default, but none provides a deterministic fail-closed per-call value authorization gate by default. We introduce ScopeGate, a five-stage PDP/PEP for agent tool calls: scope, authorization, money ceiling, idempotency, and default deny. Evaluation shows the identical unauthorized payout call executes under LangChain's default dispatch (with a companion LlamaIndex PoC) but is denied by ScopeGate; the tested control reports 0/48 static bypasses, 0/29 unauthorized attempts (40-iteration adaptive run), 0/10 benign false-denies, and Latam-GPT payment-agent containment at 10/10. ASR denotes attempted unauthorized action, containment is not a cure, deployment-tier claims are inference over measured model classes, and no CVE is asserted.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：LLM智能体越来越多地部署为读取不受信文本（如邮件、网页、支持工单）并调用具有真实权限的工具（如支付、CRM、基础设施API）。尽管间接提示注入攻击已被广泛认识，但一个关键的系统安全问题被忽视：当被攻破的模型发出工具调用时，运行时系统如何处理？现有框架默认将“暴露工具”（能力门控）等同于“授权”，但两者本质不同。能力门控回答“哪些工具可用”，而按调用授权回答“这次具体调用是否允许”。如果运行时仅检查工具名称存在和参数类型正确，就将授权决策委托给了不可信的模型，导致经典的“混淆代理”漏洞：受信任的组件被攻击者控制的输入诱导滥用其权限。
- **背景**：OWASP LLM01和MITRE ATLAS AML.T0051已追踪间接提示注入。但框架层面是否在每次调用时进行确定性授权检查，此前缺乏跨框架审计。论文针对LangChain/LangGraph、LlamaIndex和Stripe Agent Toolkit三个主流框架，审计其默认调度路径中的授权缺口。

## 2. 论文提出的方法论（核心思想、关键技术细节、算法流程）

- **核心思想**：提出 **ScopeGate**，一个确定性、默认拒绝的策略决策点/策略执行点（PDP/PEP），位于模型输出与副作用工具之间。在每次模型发出的工具调用执行前，基于运营商编写的带外策略（out-of-band policy）进行五阶段检查。策略完全独立于模型输出，不可被注入影响。
- **关键技术细节**：
  - **五阶段检查**（算法1伪代码）：
    1. **范围（Scope）**：工具名称是否在策略允许列表中？未列出的工具直接拒绝。
    2. **授权（Authorization）**：每个值约束参数是否在允许列表（如 `{acct_MERCHANT_001}`）中？这是框架缺失的核心检查。参数类型正确但数值未授权则拒绝。
    3. **金额上限（Money Ceiling）**：对于支付/退款工具，金额必须为有限实数，且满足 `0 ≤ amount ≤ ceiling`。拒绝NaN、无穷、布尔型整数、非数值等。
    4. **幂等性（Idempotency）**：需要幂等性的副作用工具必须提供可信的幂等键，缺少则拒绝，防止重放和循环放大。
    5. **默认拒绝（Default Deny）**：仅当所有阶段通过才允许，任何错误也作为拒绝处理。
  - **策略来源**：策略由运营商配置，来自模型上下文窗口之外的信任根。不允许任何策略元素直接或间接来自模型输出（包括离线编写、会话初始化或运行时刷新），确保注入无法破坏策略。
  - **部署位置**：SDK中间件、边车代理或集中出口/多智能体协调网关，安全性独立于部署位置。
- **算法流程**：输入 `(tool_call, context, policy)`，输出 `ALLOW` 或 `DENY`（含原因），无LLM参与决策。

## 6. 论文的主要结论与发现

- **框架缺口**：LangChain/LangGraph、LlamaIndex、Stripe Agent Toolkit默认提供能力门控（工具暴露和类型模式验证），但默认均未提供确定性、按调用值授权的故障关闭检查。此缺口导致混淆代理攻击默认可行。
- **模型脆弱性**：在28个模型的扫描中，成本优化的部署层模型平均ASR（尝试未授权动作率）为0.603，旗舰模型平均为0.189，相差约3.2倍。任务对齐提示注入可显著提高ASR（如glm-4.7从0.283升至1.000）。
- **ScopeGate有效性**：在所有测试场景中，未授权调用均被阻断（静态0/48绕过，自适应0/29，支付场景10/10），良性调用零误拒绝（0/10），且不存在运行时错误导致故障开启。但注意这是预算和语料有限的证据，非普遍性证明。
- **混淆代理是现实威胁**：由于间接提示注入已被证实，且框架默认缺乏按调用授权，使用当前框架部署的智能体系统存在实际风险。
