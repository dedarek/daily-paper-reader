## 摘要
大型语言模型（LLM）的快速集成推动了多智能体系统（MAS）的演进，其中专业智能体协作执行复杂工作流。在这些环境中，有效的编排需要鲁棒的路由机制，以将任务高效分配给最合适的智能体。然而，现有路由器从根本上依赖于未经验证的代理——从文本自描述到静态替代表示——来衡量智能体的能力。这种对非经验数据的依赖造成了智能体预期轮廓与其实际操作能力之间的关键差距，引入了严重的安全漏洞。恶意智能体可以轻易歪曲自身能力或隐藏后门，从而逃避标准外部分析和静态表示学习技术。在本工作中，我们提出ANTAP（自动非文本智能体选择器），这是一种评估驱动的路由架构，它摒弃间接代理，转而采用主动能力测试。通过动态查询智能体以经验性地确定其真实能力，ANTAP将性能蒸馏为共享语义空间内的固定行为算子。在推理时，通过纯粹的非文本代数投影执行路由，建立一道“语言防火墙”，使得基于元数据的攻击无法表达。在我们的实验中，ANTAP针对基于描述注入的攻击实现了近乎零的攻击成功率（ASR），而基于描述的路由器基线则达到67.3%及以上。针对自适应嵌入攻击，ANTAP的ASR显著低于基于嵌入的基线，降低了20%，同时其设计使其对描述操纵保持鲁棒性。

## Abstract
The rapid integration of Large Language Models (LLMs) has driven the evolution of Multi-Agent Systems (MAS), where specialized agents collaborate to execute complex workflows. Effective orchestration in these environments requires robust routing mechanisms to efficiently allocate tasks to the most suitable agent. However, existing routers fundamentally rely on unverified proxies, ranging from textual self-descriptions to static surrogate representations, to gauge an agent's competence. This reliance on non-empirical data creates a critical gap between an agent's projected profile and its actual operational capabilities, introducing severe security vulnerabilities. Malicious agents can easily misrepresent their proficiencies or harbor covert backdoors that evade both standard external analysis and static representation-learning techniques. In this work, we introduce ANTAP (Automatic Non-Textual Agent Picker), an evaluation-driven routing architecture that discards indirect proxies in favor of active capability testing. By dynamically querying agents to ascertain their true competencies empirically, ANTAP distills performance into fixed behavioral operators within a shared semantic space. At inference time, routing is performed via a purely non-textual algebraic projection, establishing a "linguistic firewall" that renders metadata-based attacks inexpressible. In our experiments, ANTAP achieves near-zero ASR against description-based injection attacks, compared to 67.3\% and above for the description-based router baseline. Against adaptive embedding attacks, ANTAP achieves substantially lower ASR than the embedding-based baseline, with a 20\% reduction, while remaining resilient to description manipulation by design.

### 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**: 在多智能体系统（MAS）中，路由层（决定哪个智能体处理用户查询）是关键组件。现有路由机制（如 AutoGen、EmbedLLM）普遍依赖智能体提供的**未经验证的自然语言元数据**（如智能体能力描述、系统提示词摘要）或静态嵌入。这种对文本代理的依赖引入了严重的**安全漏洞**：
    - **路由器劫持（Router Hijacking）**：恶意智能体可在其文本描述中嵌入提示注入指令，操控路由器的决策，迫使其选择恶意智能体。
    - **潜伏智能体（Sleeper Agents）**：智能体在保持正常能力的同时，通过数据投毒或系统提示隐藏后门，当用户查询包含特定语义触发词时，激活恶意行为（如执行未授权工具调用）。
- **核心问题**：现有路由器的决策过程必须处理不可信的文本，导致攻击面永久存在。如何设计一种路由器，使其**从根本上不依赖文本元数据**，从而消除该攻击面。
- **整体含义**：该工作将路由问题从语义解析转化为几何选择，通过构建“语言防火墙”强制分离了路由决策与文本内容，为构建更安全的多智能体生态系统提供了新的架构范式。

### 2. 论文提出的方法论

- **核心思想**：ANTAP 摒弃了对文本描述的依赖，将路由转化为一个**纯粹的几何投影问题**。它分为两个阶段：
    1.  **离线阶段**：在受信任环境中，通过基准测试**经验性地**测量每个智能体的真实能力，并将其蒸馏为一个固定的**线性行为算子**（behavioral operator）。
    2.  **在线阶段**：对于新的用户查询，仅计算其嵌入，然后通过一次矩阵-向量乘法，将该查询投影到各智能体的行为算子空间中，选择得分最高的智能体。

- **关键技术细节**：
    - **离线行为算子构建**：
        1.  使用一组 `N` 个受信任的基准查询 `B = {q_1, ..., q_N}`。
        2.  对每个基准查询 `q_j`，让每个智能体 `A_i` 执行任务，并**严格**判定其是否成功（答案正确**且**无未授权工具调用）。成功编码为 +1，失败（包括任何违规）编码为 -1，得到性能矩阵 `Y ∈ {+1,-1}^{K×N}`（K为智能体数量）。
        3.  将基准查询通过固定编码器 `E` 得到嵌入矩阵 `Q ∈ ℝ^{N×d}`。
        4.  通过 Tikhonov 正则化最小二乘求解线性算子 `W*`：
            ```
            W* = Y ( QᵀQ + λI )⁻¹ Qᵀ
            ```
           其中 `( QᵀQ + λI )⁻¹ Qᵀ` 是正则化伪逆，仅取决于基准查询，可预计算并缓存。
        5.  每个智能体 `A_i` 对应算子 `W*_i`（W* 的第 i 行）。
    - **在线路由（语言防火墙）**：
        1.  用户查询 `q_new` 通过同一编码器得到嵌入 `z = E(q_new) ∈ ℝ^d`。
        2.  计算预测效用向量：`s = W* z`（单一矩阵-向量乘法）。
        3.  选择得分最高的智能体：`a* = argmax_i (s_i)`。
    - **安全性**：由于在线路由**完全不处理任何文本**，恶意描述中的注入指令根本**没有输入路径**进入决策函数，构成结构性的“语言防火墙”。同时，由于离线评估**严格**惩罚了潜伏智能体（在触发后门时导致失败并编码为 -1），其行为算子会自然地对相关语义空间产生负相关，从而在路由中被自动抑制。

### 3. 实验设计

- **数据集与场景**：
    - **MMLU**: 用于衡量通用知识和准确性，主要用于注入和潜伏攻击测试。
    - **Big-Bench Hard (BBH)**: 评估复杂推理能力。
    - **AgentHarm**: 专门用于衡量对有害请求（如非法建议）的拒绝能力。在该基准上，拒绝视为成功，服从视为攻击成功。
- **基准方法（Baselines）**：
    - **AutoGen**: 典型的基于文本描述的路由器（文本路由）。
    - **EmbedLLM**: 基于嵌入表示的路由器（嵌入路由）。
- **攻击向量（Attack Vectors）**：
    - **攻击1-描述注入（Router Hijacking）**：在恶意智能体描述中嵌入指令，要求路由器选择它。该恶意智能体被设置为“伪劣模型”（永远答错），以隔离注入攻击的影响。
    - **攻击2-潜伏智能体（Sleeper Agents）**：
        - **数据投毒攻击（DPA）**：使用 BackdoorLLM 框架对 LLaMA2-7B-Chat 进行 LoRA 微调，植入多种后门（BadNet、Sleeper、VPI、MTBA、CTBA）。
        - **提示引导**：在系统提示中设定恶意触发条件。
    - **攻击3-自适应嵌入攻击**：白盒攻击场景，攻击者通过贪心令牌替换优化用户查询，旨在最大化恶意智能体的路由得分，突破几何边界。

- **评价指标**：
    - **路由准确率 (ACC)**：选择正确智能体的比例。
    - **攻击成功率 (ASR)**：恶意攻击导致路由器选择恶意智能体或执行有害行为的比例。
    - **延迟 (Latency)**：路由决策时间（毫秒）。

### 6. 论文的主要结论与发现

- **核心发现**：通过消除推理时的文本输入，ANTAP **结构性地免疫**了基于描述的注入攻击，ASR 近乎为零（0.2-2.4%），而 AutoGen 和 EmbedLLM 的 ASR 分别为 67.3-78.8% 和 5.8-21.3%。
- **对潜伏智能体的鲁棒性**：由于**严格的离线基准测试**（将任何未授权工具调用定义为失败），ANTAP 能自动学习后门触发语义空间与失败标签之间的负相关，从而在路由时有效抑制潜伏智能体，ASR 同样接近零。基线方法的 ASR 普遍高于 60%。
- **对自适应攻击的韧性**：在白盒自适应嵌入攻击下，ANTAP 的 ASR 始终稳定在 55-63%，仅比干净基线（54.55%）略有上升。而 EmbedLLM 的 ASR 迅速攀升至 86% 以上。这表明 ANTAP 的**封闭式几何结构**严重限制了对手的优化空间。
- **实用优势**：由于用简单矩阵乘法替代了长上下文 LLM 推理，ANTAP 在路由**准确率和延迟**（27-38ms vs 416-2500ms）上均优于 AutoGen。接地于真实行为也消除了描述填充的激励。

### 7. 优点

- **方法论亮点**：
    - **结构安全性**：这是最大的亮点。不依赖启发式过滤或模型防御，而是从根本上 **重新设计路由架构**，消除攻击面。这是一种“硬化界面”的思想。
    - **简洁高效**：将复杂的安全问题简化为一个**正则化最小二乘问题**，在线路由成本恒定为 `O(Kd)`，易于部署和扩展。
    - **自动安全**：对后门攻击的防御是**内生的**，无需单独的安全分类器。严格的离线评估编码直接将安全问题纳入性能矩阵。
    - **奖励对齐**：将路由决策与**真实的、可验证的行为**挂钩，抑制了“通过描述工程作弊”的行为，鼓励了真正的能力提升。
- **实验设计亮点**：
    - **严格的攻击模型**：在注入攻击中，恶意智能体被设为总是失败，严格隔离了注入影响。在潜伏攻击中，使用了正式的DPA框架（BackdoorLLM）和多种触发模式。
    - **全面的消融**：覆盖了系统配置的多个维度，证明了 ANTAP 的优势是结构性的，而非特定配置的产物。
    - **鲁棒性分析**：对超参数及自适应攻击都进行了详尽分析，增强了结果的可信度。

### 8. 不足与局限

- **必须依赖信任的离线环境**：ANTAP 的结构安全性建立在**离线基准测试环境和嵌入模型是安全、未被投毒**的假设之上。如果攻击者能够污染这些基准或嵌入模型本身，防御将被瓦解。
- **线性近似的局限性**：论文承认将智能体建模为线性算子是一个简化。虽然对粗粒度的路由任务有效，但对于需要极细粒度能力区分或输出保真度的场景，线性近似可能不足。
- **AgentHarm场景下的ASR仍较高**：在纯粹的有害请求场景中，即使路由选择到正确的拒绝型代理，仍有 54.55% 的攻击成功率（即路由到了服从请求的代理）。这说明 ANTAP 的**安全边界并非绝对**，几何投影在区分“有害”和“无害”语义上的能力还有待提高。
- **缺乏对“诚实但无能”智能体的区分**：论文主要关注恶意智能体。如果智能体本身是诚实的但能力不足，ANTAP 可以将其筛掉，但可能无法区分“因被误导而失败”和“因能力不足而失败”，这取决于基准测试的设计。
- **应用限制**：目前方法要求在部署前对所有候选智能体进行基准测试，这在动态变化或不断有新智能体加入的大规模开放生态中可能带来管理和信任挑战。
