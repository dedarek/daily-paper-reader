## 摘要
智能体越来越多地代表用户生成SQL、编排管道并自动化数据分析。尽管最近的工作提高了查询的正确性，但正确性不等于安全性。一个查询可能在语义上有效，但违反了管理数据如何组合和发布的监管、隐私或业务约束。我们认为，执行此类约束本质上是数据基础设施的问题。本文介绍了数据流控制（DFC），这是一个框架，用于在DBMS查询中声明性地指定并保证对元组级数据流的策略执行。一个关键挑战是定义一种不依赖于优化器但在大规模执行时仍然高效的政策语言。我们将数据安全形式化为对溯源单项式的聚合谓词，并提出了Passant，一个可移植的查询重写层，无需物化溯源即可执行DFC策略。在五个DBMS引擎（DuckDB、Umbra、PostgreSQL、DataFusion和SQLServer）上，Passant实现了约0%的开销，并且性能比替代方案高出几个数量级。因此，数据流控制是将数据安全从提示和事后检查转移到数据基础设施的第一步。数据流控制已在https://github.com/dataflowcontrol/data-flow-control开源提供。

## Abstract
Agents increasingly generate SQL, orchestrate pipelines, and automate data analysis on behalf of users. While recent work improves query correctness, correctness is not safety. A query may be semantically valid yet violate regulatory, privacy, or business constraints that govern how data may be combined and released. We argue that enforcing such constraints is fundamentally a data infrastructure problem.   This paper introduces Data Flow Control (DFC), a framework to declaratively specify and guarantee policy enforcement over tuple-level data flows within a DBMS query. A key challenge is defining a policy language that is optimizer-invariant yet efficient to enforce at scale. We formalize data safety as aggregate predicates over provenance monomials and present Passant, a portable query rewriting layer that enforces DFC policies without materializing provenance. Across five DBMS engines -- DuckDB, Umbra, PostgreSQL, DataFusion, and SQLServer -- Passant achieves ~0% overhead and outperforms alternatives by orders of magnitude. As a result, Data Flow Control is the first step towards moving data safety from prompts and post-hoc checks into the data infrastructure. Data Flow Control is available open source at https://github.com/dataflowcontrol/data-flow-control.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：AI智能体（如TaxAgent）越来越多地代表用户生成SQL、编排ETL管道并自动化数据分析。然而，**查询的语义正确性并不等同于安全性**——一个语法和语义正确的查询仍可能违反监管法规（如税务法）、隐私约束（如用户数据不得单独泄露）或业务规则（如数据不得组合）。现有的安全机制（访问控制、完整性约束、数据溯源审计）都只能事后或粗粒度地处理，而不能在查询执行过程中主动、确定性地禁止不合规的数据流。
- **整体含义**：作者主张**数据安全本质上是一个基础设施问题**，而不是靠提示工程或后置检查。为此提出**数据流控制（Data Flow Control, DFC）**框架，将安全策略嵌入DBMS内部，实现对元组级数据流的声明性、高效强制执行。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将数据安全策略形式化为**关于查询输出元组所对应的溯源单项式的聚合谓词**。策略定义在溯源单项式（provenance monomial）上，从而与物理查询计划无关（优化器不变性）。通过查询重写将策略评估融入查询执行，避免物化完整的溯源多项式。
- **关键技术细节**：
    - **PGN策略语言**：使用`SOURCE`、`SINK`、`DIMENSION`、`CONSTRAINT`、`ON FAIL`等子句声明性地定义策略。支持聚合条件、外部维度表、输出引用、必选源、关系级分辨率等扩展。
    - **Passant查询重写层**：实现三种重写策略：
        - **Naive (Post-process)**：先用逻辑或物理溯源系统物化多项式，再后处理计算策略——开销巨大。
        - **Partial-Push**：将策略聚合下推到最终连接之前，减少中间结果大小，但仍需额外扫描基表。
        - **Full-Push**：利用半环聚合的性质，在查询执行过程中“捎带”计算策略所需的半环聚合，完全避免溯源物化。适用于所有半环聚合（count、sum、max、distinct等）。关键步骤：①自顶向下分析策略聚合，在源关系扫描时初始化半环元素；②自底向上通过连接（乘法）和聚合（加法）传播和计算部分聚合。
    - **扩展支持**：处理SQL-92中的外连接、半连接、排序/限制等；支持非`⊥`的分辨函数（如转换）和关系级分辨率；针对模板化策略的**代数优化**（单调阈值合并、精确匹配简化、排除合并）；针对自连接的**对称策略优化**（将阶乘次评估降为线性）。
- **算法流程**（文字说明）：
    1. 输入：用户查询Q、策略集合P。
    2. 静态分析：判断策略是否适用（源表匹配）。
    3. 选择重写策略：如果策略聚合均为半环，则使用Full-Push；否则使用Partial-Push或回退。
    4. 执行重写后的查询：在基表扫描时给每个元组添加半环标签（如计数初始1、属性值等）；在连接处标签相乘；在聚合处标签相加；最终在HAVING阶段评估策略条件并过滤/转换输出元组。

### 6. 论文的主要结论与发现

- **Full-Push跨五大DBMS引擎实现接近零开销**（大多数查询<5%），且多数情况下比No Policy更快。
- **Partial-Push和Logical有明显开销**（Logical可达数千%减速），物理方法在输出元组多且贡献少时开销大。
- **策略数量扩展时，利用代数优化（单调阈值合并）可将评估时间从线性降低到常数**（百万级策略下18ms vs 原始需要遍历）。
- **对称自连接策略的优化将阶乘级评估降为线性**（1000次自连接时从>1m次评估降为1k次）。
- **在状态机应用上，Full-Push与GPT-5.2同样正确，但速度快两个数量级**。
- **结论**：数据流控制应作为数据基础设施的一部分，Passant高效、可移植地实现了这一目标。

### 7. 优点

- **方法创新**：首次将数据安全策略形式化为溯源单项式上的聚合谓词，实现优化器不变性；提出Full-Push重写，避免溯源物化。
- **实用性**：PGN语言简洁，兼容SQL-92，支持外部维度、LLM调用、多种分辨率函数；Passant可运行在5种主流DBMS上，无需修改数据库内核。
- **实验设计全面**：覆盖多种引擎、多种查询类型、多种策略复杂度，并进行了扩展性（策略数/自连接数）和工程优化验证。
- **性能优势显著**：Full-Push近乎零开销，远远优于现有溯源方法和LLM审查，使“默认安全”成为可能。
- **开源**：代码公开，便于复现和进一步研究。

### 8. 不足与局限

- **适用范围有限**：当前仅支持单调SQL-92查询（SPJUA + 外连接等），非单调操作（差集、递归、窗口函数）的语义未定义，需要后续工作。
- **未考虑多查询/会话级策略**：论文专注于单查询执行，跨查询轨迹的agent策略（如Prompt Injection防御）留作未来。
- **GPU/LLM部分未深入**：LLM作为策略中可选的维度仅举例，并未系统评估LLM作为策略函数时的开销和准确性。
- **实验偏差风险**：所有实验在单机或小型RDS上运行，未测试大规模分布式集群下Full-Push的扩展性；策略均假设不删除元组（最坏情况），实际可能有剪枝加速，但未报告剪枝比例。
- **策略语言表达力限制**：多源非半环聚合（如`max(T.x+S.y)`）无法全推送，需回退到Partial-Push，论文仅简单提及。
- **模板化策略优化**：仅支持单属性比较谓词类模板，更复杂的带析取条件的模板优化可能遇到NP-hard问题，尚未解决。
- **用户研究缺失**：未进行PGN的可用性/用户理解度实验，仅推测SQL从业人员熟悉。
