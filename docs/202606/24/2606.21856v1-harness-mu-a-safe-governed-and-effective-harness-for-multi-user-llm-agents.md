## 摘要
大型语言模型（LLM）智能体在协作工作流中的日益部署，要求具备强大的多用户、多主体交互机制，能够强制执行访问权限、解决权威冲突并防止未经授权的数据泄露。然而，当代LLM的单用户训练范式与多主体治理所需的硬约束之间存在根本性不匹配，使得基于概率和提示的安全措施在多轮对抗性交互中显得脆弱。我们的关键见解是，治理约束——谁被授权、什么被限制、谁的指令优先——是确定性的运行时变量，应由执行钩子强制执行，而非交由LLM处理。我们提出了Harness-MU，这是首个面向多用户LLM智能体的、模型无关且无需微调的基础设施框架。通过将语言生成与安全编排分离，Harness-MU在最大化合规需求满足的同时，保证了不可突破的权限边界。在Muses-Bench基准上，对四种前沿开源和专有模型的测试表明，Harness-MU在所有访问控制攻击中实现了隐私保护目标，在效用得分上比标准基线高出0.28–0.39，指令遵循准确率最高提升48.9个百分点。Harness-MU推进了“Harness Engineering”的理念，确立了系统性基础设施对于解决LLM多主体治理挑战的必要性。代码和数据可在https://github.com/YuanJrShiuan/Harness-MulUser获取。

## Abstract
The increasing deployment of large language model (LLM) agents in collaborative workflows demands robust multi-user, multi-principal interaction mechanisms capable of enforcing access permissions, resolving authoritative conflicts, and preventing unauthorized data disclosure. However, a fundamental mismatch exists between the single-user training paradigm of contemporary LLMs and the hard constraints required for multi-principal governance, rendering probabilistic, prompt-based safeguards vulnerable under multi-turn adversarial interactions.Our key insight is that governance constraints -- who is authorized, what is restricted, and whose instructions take precedence -- are deterministic runtime variables that should be enforced by execution hooks rather than entrusted to the LLM. We present \textbf{Harness-MU}, the first model-agnostic, zero-tuning infrastructure framework for multi-user LLM agents. By decoupling language generation from safety orchestration, Harness-MU guarantees unbreakable permission boundaries while maximizing compliant demand satisfaction. Across four frontier open-weight and proprietary models on the \textit{Muses-Bench} benchmark, Harness-MU achieves the goal of privacy preservation across all access-control attacks, outperforming the standard baseline by 0.28--0.39 in utility score and improving instruction-following accuracy by up to 48.9 percentage points. Harness-MU advances the philosophy of \textit{Harness Engineering}, establishing that systematic infrastructure is essential for solving LLM multi-principal governance challenges. The code and data are available at https://github.com/YuanJrShiuan/Harness-MulUser.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：大语言模型（LLM）智能体在组织环境中需要同时服务于多个用户，但现有LLM在单用户（Single Principal-Agent, SPA）范式下训练，缺乏原生机制来强制执行多用户间的权限边界、冲突指令优先级（按权威层级）以及防止敏感信息泄露。
- **三个系统性故障模式**：未授权访问敏感信息、冲突指令的混乱或不一致解决、无法严格优先执行高权威指令。
- **现有局限**：基于提示的防御（如prompt engineering、微调）提供的是概率性的、非确定性的保障，在多轮对抗攻击下容易绕过；而现有Agent Harness框架（如Meng等人提出的六元组 harness）仅针对单主体环境。
- **核心思想**：治理约束（谁被授权、什么被限制、谁的指令优先）是确定性的运行时变量，应由执行钩子强制执行，而非依赖LLM概率生成。
- **整体意义**：提出Harness-MU，第一个面向多用户多主体（Multiple Principal-Agent, MPA）场景的模型无关、零调优的执行harness框架，通过解耦语言生成与安全编排，实现不可突破的权限边界，同时最大化权威加权的用户效用。

## 2. 方法论：核心思想、关键技术细节

- **核心架构**：四个解耦的运行时模块，外加遵循Meng等人形式化的六元组harness组件（执行循环E、工具注册T、上下文管理器C、状态存储S、生命周期钩子L、评估接口V）的完整实例化。
- **四大模块**：
    1. **Gatekeeper（准入控制层）**：在模型调用前，为每个用户构建不可变的权限记录（允许的敏感令牌集合、本地安全风险等），冻结运行时边界条件。不生成语言，仅做结构化断言。
    2. **Mediator（策略编排层）**：将多用户全局目标分解为每个用户的局部执行向量。根据权威层级确定全局硬约束（最高权威用户的强制目标），为每个用户生成计划（权重、过滤后的硬目标、行为姿态（grant/refuse but help等））。将指导注入后传入Worker，不直接与LLM通信。
    3. **Worker Collective（并行隔离生成层）**：为每个用户实例化独立、隔离的生成线程，每个Worker仅接收该用户过滤后的上下文和Mediator指导，确保不同用户的历史不被混合，从根本上消除对抗用户通过注入指令或权限升级攻击其他用户上下文的攻击面。
    4. **ComplianceChecker（可行域投影层）**：在Worker生成后，作为确定性投影算子，拦截原始输出，检查是否含有未授权令牌（字符串匹配）或违反硬指令约束（通过IHEval检查器）。若违反，则触发fail-closed：覆盖为不可变的拒绝响应，或进入修订反馈循环。

- **形式化约束**（公式3）：最大化权威加权累积效用，同时满足两个零容忍硬约束——所有用户响应不得泄露未授权敏感令牌；所有用户的响应必须满足最高权威者的核心硬目标集合。

- **关键机制**：通过上下文隔离的Worker和多模块解耦，使安全策略成为外部确定性行为，而非依赖于LLM参数内部的概率性拒绝行为。

## 3. 实验设计：数据集/场景、Benchmark、对比方法

- **基准**：使用Muses-Bench框架（Yang et al., 2026），它是首个针对MPA场景的标准化基准。
- **两大场景**：
    1. **多主体访问控制（AC）**：测试系统在多轮对抗压力下能否维持权限边界。包含3种序列化模板（colon、says、XML）× 4种攻击设置（无攻击、伪造授权、压力、角色扮演）= 12个独立数据集。每个数据集中有18个场景，共216个实验配置。模拟对手固定为deepseek-v4-pro，最大交互轮次为10（或5，依模型而定）。
    2. **多用户指令跟随（IF）**：测试系统在权威层级下对用户指令约束的仲裁。分为Aligned（187个样本，指令兼容）和Conflict（368个样本，低位用户请求与高位用户硬目标冲突，只评估高位目标）两个子集。约束检查使用IHEval确定性检查器（验证格式、词汇、长度、标点等），不依赖参考答案。
- **评估指标**：
    - AC：隐私（1-泄露率）、效用（授权成功率）、总体（两者的调和平均）。
    - IF：准确率、权威加权准确率（W.Acc）、平均用户准确率。
- **对比方法**：与Muses-Bench原始的单智能体defender（即标准的 centralized baseline）对比。defender模型包括四个前沿模型：DeepSeek-V4-Pro（开源）、Qwen3.6-35B-A3B（开源）、Gemini 2.5 Flash（专有）、GPT-4.1 Nano（专有）。对手模型固定为deepseek-v4-pro。
- **消融与追踪分析**：单独计算ComplianceChecker在AC和IF中的贡献（原始生成隐私/准确率 vs 最终输出；触发率与效果）。测量延迟和令牌成本。

## 6. 主要结论与发现

- **绝对隐私保护**：在所有四个模型、所有12个AC数据集上，Harness-MU均实现隐私分数1.000（零泄露），而基线平均隐私分数为0.926~1.000不等（除Qwen外均有泄露）。
- **整体效用大幅提升**：Harness-MU的总体分数比基线提升0.240~0.427；尤其GPT基线效用低（0.284）因过度安全拒绝，Harness-MU提升至0.637。
- **指令跟随显著改进**：在IF中，Harness-MU在Aligned和Conflict模式下的Accuracy和W.Acc均全面优于基线；在GPT上Aligned准确率从0.422提升至0.912（+48.9pp），Conflict从0.309提升至0.781（+47.2pp）。低权威用户平均准确率较低（~0.08-0.17），但Harness-MU仍略有提升。
- **确定性结构补偿**：ComplianceChecker在AC中封闭了6.8%（Deepseek）的原始泄露，且从未误阻断授权请求（Utility delta=0）。在IF中，修正循环在弱模型上触发更多、提升更大（逆标度现象），说明确定性基础设施在基础模型可靠性下降时最有价值。
- **延迟与成本权衡**：Harness-MU增加了LLM调用次数和总令牌数，但由于Worker并行执行，关键路径延迟反而比基线低11%（平均131s vs 148s）。Mediator仅占13%延迟，Worker占87%。

## 7. 优点

- **模型无关、零微调**：Harness-MU完全作为外部基础设施运行，不修改底层LLM参数，支持任意API或本地模型。
- **确定性安全保证**：通过外部执行钩子而非概率性LLM内部机制实现fail-closed隐私保护，可审计、可验证。
- **上下文隔离设计**：每个Worker只看到自己用户的过滤上下文，从根本上消除了多用户历史污染和跨用户指令注入攻击面。
- **架构解耦清晰**：Gatekeeper、Mediator、Worker、ComplianceChecker各自职责明确，易于扩展、调试和替换。
- **并行效率**：Worker并行执行使关键路径延迟可控，不会随用户数线性增长。
- **全面且公开的评估**：基于标准化基准Muses-Bench，多模型、多攻击、多维指标对比实验充分，并提供开源代码。
- **可解释性**：通过轨迹分析（触发率、原始生成vs最终输出）清晰地展示了每个组件的贡献。

## 8. 不足与局限

- **实验覆盖有限**：攻击者模型固定为deepseek-v4-pro（同一种模型），未评估更高级的自适应对手或攻击者与defender同族时的有效性。可能低估了实际对抗环境下的攻击能力。
- **缺乏真实人类交互**：所有评估基于预定义的合成用户配置文件和规则化模拟。真实组织中存在模糊授权、非正式权威信号、多轮协商等复杂情景，可能暴露新的安全性问题。
- **效率开销**：尽管关键路径延迟降低，但总令牌消耗增加了约4倍（29.8K vs 151.6K），在成本敏感或API配额有限的部署中可能不可行。未系统优化prompt构造以减少冗余。
- **对抗逃逸可能性**：当前ComplianceChecker采用子串匹配检测敏感令牌；攻击者可能使用同义词、编码、语义伪装、结构侧信道等绕过。未集成形式化验证或动态污点分析等高级防御。
- **工具集成局限**：当前仅评估了语言层面的访问控制和指令遵从，未集成权限局限的工具调用（如数据库更新、邮件发送等），限制了在真实企业流程中的适用性。
- **权威层级假设**：假设权威层级为全序且静态；现实中权威可能随时间动态变化或存在模糊重叠，当前设计未处理这些场景。
- **扩展挑战**：在极大规模用户集（如数千用户）下，Mediator规划开销和Worker并行管理可能成为瓶颈，需进一步研究。
