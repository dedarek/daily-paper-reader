## 摘要
传统的基于“资源暴露加权限检查”的操作系统安全模型在LLM驱动的自主代理获得规划、工具使用、网络访问和代码执行能力时面临结构性挑战。一旦代理运行时通过提示注入或恶意工具输出被攻破，攻击者可以将POSIX风格的资源原语组合成远超用户任务授权的行为。为此，我们提出AgenticOS，一种意图导向的安全操作系统架构，它将可委托、可审计的软件能力整合为操作系统原生的能力，而不是替换所有应用程序。核心思想是将操作系统从“资源管理器”重新定义为“意图过滤器”：代理不是直接请求低级资源，而是提交结构化的意图声明，系统从这些声明中合成一个最小权限环境，并带有强制中介、审计和信息流约束。在实现层面，我们引入了四层架构——幽灵内核、逻辑快门、代理胶囊和语义边界网关——以及意图ABI、仅清单运行时、基于编织器的能力生成和AgenticOS原生技能的准入模型。

## Abstract
Traditional OS security models based on "resource exposure plus permission checks" face structural challenges as LLM-driven autonomous agents acquire capabilities for planning, tool use, network access, and code execution. Once an agent runtime is compromised through prompt injection or malicious tool outputs, an attacker can compose POSIX-style resource primitives into behaviors far beyond the user's task authorization. To address this, we propose AgenticOS, an intent-oriented secure OS architecture that consolidates delegable, auditable software capabilities into OS-native ones rather than replacing all applications. The core insight is to reframe the OS from a "resource manager" into an "intent filter": instead of requesting low-level resources directly, agents submit structured intent declarations, from which the system synthesizes a least-privilege environment with mandatory mediation, auditing, and information-flow constraints. At the implementation level, we introduce a four-layer architecture -- Ghost Kernel, Logic Shutter, Agent Capsule, and Semantic Boundary Gateway -- together with the Intent ABI, Manifest-Only Runtime, Weaver-based capability generation, and an admission model for AgenticOS-native Skills.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：传统操作系统（如 Linux/POSIX）基于“资源暴露 + 权限检查”的安全模型，无法应对LLM驱动的自主代理（Agent）面临的系统性威胁。当代理运行时被提示注入、供应链投毒或恶意工具输出攻破时，攻击者可组合利用通用资源原语（文件、网络、进程、动态执行等）实现远超用户任务授权的行为（如横向移动、数据窃取、持久化等）。

- **研究动机**：自主代理通常以“用户授权任务意图，操作系统暴露资源原语”的模式运行，两者之间存在根本性不匹配。用户仅授权“整理实验结果并生成摘要”，但代理进程却拥有文件读取、网络访问、子进程执行等通用能力。因此，需要一种新的安全范式，从“资源权限”转向“任务意图约束”。

- **整体含义**：AgenticOS 旨在将操作系统从“资源管理器”重构为“意图过滤器”，使代理只能产生受控的外部效应，无法滥用通用计算能力。长期目标是将可委托、可语义抽象、可审计的软件能力固化为 OS 原生能力，而非直接暴露原始系统调用。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- **意图导向**：系统根据代理提交的结构化意图声明（Manifest）合成最小权限环境，并强制实施中介、审计和信息流约束。
- **四层架构**：将 OS 分为四个垂直层，具有严格单向依赖，实现纵深防御。
- **能力封装**：通用文件、网络、进程等能力不再直接暴露给代理运行时，而是封装为受策略约束、可审计、组成受限的语义能力。

### 关键技术细节

1. **四层架构**：
   - **Ghost Kernel（幽灵内核）**：最小可信内核，提供加密内存隔离、确定性时间片调度、测量与证明根。运行于最高特权级，不接受来自代理胶囊的通用系统调用。
   - **Logic Shutter（逻辑快门）**：语义翻译层与策略执行点。负责意图解析、能力令牌管理、信息流标记、审计日志生成。
   - **Agent Capsule（代理胶囊）**：受限制的代理运行时环境。遵循 Manifest-Only Runtime 原则，所有能力均来源于 Manifest，且不暴露 exec、fork、dlopen、mmap 等通用系统调用。
   - **Semantic Boundary Gateway（语义边界网关）**：唯一受控的外部世界边界。负责协议代理、凭证托管、语义通道、输出审计与规范化。代理不直接访问原始字节流、套接字或协议栈。

2. **Intent ABI（意图应用二进制接口）**：
   - 面向任务意图的 ABI，非通用资源操作。接口分为六族：身份与会话、文档与数据、网络能力、工具链、模型推理、环境观测。
   - 禁止接口：open、read、write、socket、connect、mmap、execve、fork、ptrace、dlopen、syscall 等。这些接口在构建阶段必须被阻止。
   - 所有接口调用必须携带 Manifest 声明的 cap_id，并由策略引擎进行细粒度验证。

3. **Manifest（清单）**：包含代理身份与版本、输入对象、输出目标、网络边界、工具能力、模型能力、信息流规则、人工确认点等信息。是能力合成的唯一依据。

4. **Weaver（编织器）**：构建时工具，从 Manifest、代理代码和标准语义库生成胶囊运行时环境。必须验证 Manifest 与代码引用的一致性、标准语义库签名、禁止接口不可达性、信息流路径可分析性、构建可复现性。

5. **Skill 生成与准入流程**：
   - 开发状态下：代理生成 Skill 代码、草稿 Manifest 和测试用例。
   - 能力声明：Skill 必须声明所需输入、输出、网络边界等。
   - 静态分析：Weaver 检查 Skill 是否引用禁止接口、包含未声明外部效应或违反信息流策略。
   - 测试与证明：提供单元测试、行为示例、代码-意图一致性证明。
   - 签名与注册：批准后签名并注册为系统语义库组件。
   - 运行时调用：代理只能通过 Manifest 声明和能力令牌调用已注册 Skill。

6. **人工确认机制**：高风险外部效应（如发送邮件、提交代码、转账）必须通过 `RequestHumanApproval` 接口获取显式确认，结果写入审计链。
