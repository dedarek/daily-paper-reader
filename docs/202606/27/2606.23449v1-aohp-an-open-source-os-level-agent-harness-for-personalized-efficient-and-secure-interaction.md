## 摘要
AI智能体正在推动一种新的软件范式，能够自主调用工具、提取信息、管理内存并完成跨应用和数据源的任务。然而，现有的终端操作系统大多是为以应用为中心的工作流设计的，对AI智能体的原生支持很少。这种不匹配限制了智能体的广泛采用，并在传统系统上运行智能体时导致执行开销和安全风险。虽然智能体原生操作系统的概念正在兴起，但研究界缺乏一个开放的测试平台来探索智能体中介交互所需的架构原语。我们提出了AOHP（Android开放框架项目），一个基于Android开源项目（AOSP）构建的OS级智能体框架。AOHP的核心设计原则是将智能体视为一流的操作系统参与者，实现自适应用户界面和智能体友好的运行时环境。AOHP保留了成熟的Android软硬件生态系统，同时引入了三个面向智能体的系统机制：个性化服务组合、高效智能体接口和安全信息流。基于覆盖OS智能体关键能力的挑战性任务的初步实验，AOHP在任务完成率（+21.12%）、执行成本（-51.55% token成本）和安全策略合规方面显示出明显优势。

## Abstract
AI agents are driving a new software paradigm, with the ability to autonomously call tools, extract information, manage memory, and complete tasks that span applications and data sources. Most existing end-user operating systems, however, are designed for application-centric workflows and offer little native support for AI agents. This mismatch limits the wider adoption of agents and leads to execution overhead and safety risks when running agents on conventional systems. While the concept of agent-native operating systems is emerging, the research community lacks an open testbed to explore the architectural primitives desired for agent-mediated interaction. We present AOHP (Android Open Harness Project), an OS-level agent harness built on the Android Open Source Project (AOSP). The core design principle of AOHP is to treat agents as first-class OS actors, enabling adaptive user interfaces and agent-friendly runtime environments. AOHP preserves the mature Android software and hardware ecosystem while introducing three agent-oriented system mechanisms: personalized service composition, efficient agent interfaces, and secure information flow. Based on preliminary experiments on challenging tasks covering key capabilities of OS agents, AOHP shows clear advantages in task completion (+21.12% completion rate), execution cost (-51.55% token cost), and security-policy compliance.

## 1. 论文的核心问题与整体含义（研究动机和背景）
现有终端操作系统（如 Android、Windows）本质上是**以应用为中心**设计的：用户界面由开发人员固定，应用生命周期与显示屏强耦合，权限机制局限于应用边界，无法追踪跨应用的数据流。然而，随着 AI 智能体（agent）的兴起，用户希望智能体能自主调用工具、跨应用提取信息、管理内存并完成多步骤任务。这种“应用中心”的假设与“智能体中介”的工作流之间存在严重的架构不匹配，导致执行效率低（频繁的 GUI 导航、冗余的截图解析）、安全风险高（敏感数据易在 agent 上下文中暴露）、且难以实现个性化服务组合。因此，**需要从操作系统层面提供对智能体的原生支持**，AOHP 正是为此而设计的。

## 2. 论文提出的方法论：核心思想、关键技术细节
### 核心思想
将智能体视为操作系统的**一等公民**，在保持 Android 软硬件生态兼容的基础上，通过系统层重构实现三个目标：  
- **个性化服务组合**：生成任务级入口，聚合跨应用能力。  
- **高效智能体接口**：减少视觉处理开销、支持并行后台执行、结构化交互。  
- **安全信息流**：默认隔离敏感数据，通过污点追踪和可信执行控制流出。

### 关键技术细节
1. **个性化服务组合**  
   - **生成式服务入口**：由任务模式（task schema）、服务图（service graph）和呈现策略组成。OS 代理自动发现并组合 API、CLI、GUI 三种通道的能力，构建针对用户意图的交互界面。  
   - **能力发现与组合**：每个能力带有输入/输出模式、前置条件、副作用和策略标签，代理按权限约束组合。  
   - **跨服务个性化**：系统内存分为持久化配置文件内存、任务本地内存和敏感内存，使偏好可跨应用共享且受审计。

2. **高效智能体接口**  
   - **并行后台交互**：引入轻量级虚拟显示屏，解耦应用生命周期与物理显示，允许 agent 在后台运行等待型或独立工作流。  
   - **Agent-aware UI 增强**：将传统 GUI 抽象为结构化表示（降低冗余、丰富语义），同时保留渲染 GUI 回退。  
   - **原生沙箱运行时**：提供 OS 管理的独立执行环境（可创建、重置），agent 可在其中执行代码、处理数据，无需占用 agent 上下文。  
   - **统一文件快捷方式**：将文件视为 OS 边界的一等任务对象，GUI 操作直接影响存储时反射为结构化文件观测，agent 可直接访问或通过系统分享流传递。  
   - **事件流抽象**：支持订阅两类连续数据源：① 动态通知捕获（Toast、推送、弹窗等瞬态消息）；② 传感器数据流（加速计、麦克风等）。agent 无需轮询即可响应系统/环境变化。

3. **安全信息流**  
   - **策略执行**：基于运行时数据使用（数据源、目的、动作敏感性、批准状态）评估权限，而非静态应用身份。  
   - **敏感源清理**：在敏感内容进入 agent 上下文前，替换为类型化占位符（如 `<payment-card:uuid>`），真实值存储在可信数据保险库中。  
   - **可信保险库与执行**：agent 通过占位符向可信保险库提交操作请求，保险库检查策略、获取用户批准后执行（如格式化、比较、填充字段），敏感结果仍返回占位符。  
   - **数据流污点追踪**：敏感数据关联污点元数据，随复制、转换、组合、传输传播，在系统出口检查并记录审计轨迹。
