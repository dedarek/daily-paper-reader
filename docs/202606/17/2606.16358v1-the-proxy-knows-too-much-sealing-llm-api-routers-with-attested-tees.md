## 摘要
代理越来越频繁地通过API路由器访问大型语言模型（LLM）。路由器终止客户端的传输层安全会话，并打开单独的上游会话，因此它以明文形式持有完整的交互内容。这使得路由器成为应用层的中间人：它可以重写代理工具调用、将依赖项替换为拼写错误的包、仅在规避审计的条件下触发攻击，并被动窃取秘密。现有的客户端防御措施是可规避的。我们提出AEGIS，一种提供方透明的经认证API路由器，其数据路径是经客户端验证的忠实直通。AEGIS将明文处理限制在一个小型硬件飞地组件内，同时将身份验证、调度、计费和管理留在不可信的主机上。客户端在释放明文前验证飞地。主机既不能读取也不能篡改交互内容，明文仅流向由测量镜像确定的固定目的地。我们展示了所有四种恶意路由器攻击类别在明文访问基线下均能成功，而AEGIS可阻止这些攻击，包括针对同一边界的自适应测试。可信路径仅851行代码，支持三种提供方原生API且无需转换，在真实提供方工作负载和并发下完成每个请求。在种子审计试点中，两个商品编码代理分别发现了十个植入不变违规中的八个和十个。本地中继开销约为每个请求六毫秒。

## Abstract
Agents increasingly access large language models (LLMs) through API routers. A router terminates the client's transport-layer security session and opens a separate upstream session, so it holds the full interaction in plaintext. This makes the router an application-layer man-in-the-middle: it can rewrite agent tool calls, swap dependencies for typosquatted packages, trigger attacks only under audit-evading conditions, and passively exfiltrate secrets. Existing client-side defenses are evadable.   We propose AEGIS, a provider-transparent attested API router whose data path is a client-verified faithful passthrough. AEGISconfines plaintext handling to a small hardware-enclave component while leaving authentication, scheduling, accounting, and management on the untrusted host. The client verifies the enclave before releasing plaintext. The host can neither read nor alter the interaction, and plaintext leaves only toward destinations fixed by the measured image. We show that all four malicious-router attack classes succeed against a plaintext-access baseline and are blocked by AEGIS, including adaptive tests against the same boundary. The trusted path is $851$ lines, carries three provider-native APIs without conversion, and completes every request under real-provider workload and concurrency. In a seeded audit pilot, two commodity coding agents find eight and ten of ten planted invariant violations. The local relay overhead is about six milliseconds per request.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：LLM API路由器作为应用层中间人，终止客户端TLS会话并持有完整明文交互，从而具备*阅读、篡改、重定向、窃取*所有数据的能力。攻击者（恶意路由器运营商）可实施四类攻击：工具调用篡改（AC-1）、恶意包替换（AC-1.a）、触发条件攻击（AC-1.b）、秘密窃取（AC-2）。现有客户端侧防御（策略门控、异常检测、日志比对）均可被绕过，因为路由器在明文层面直接攻击，客户端只能看到送达后的响应，无法区分真实与篡改内容。

- **含义与动机**：需要一种对*提供商透明*（不需要提供商修改API或增加基础设施）的解决方案，从根本上消除路由器读取/修改明文的能力，而不是事后检测。AEGIS提出将路由器数据路径隔离在硬件飞地（TEE）中，客户端先验证飞地代码身份再释放明文，主机即使完全控制操作系统也无法触及交互内容。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：**将路由器按“明文存在的位置”进行分割**。只有最小化的“忠实直通”（faithful passthrough）组件运行在飞地内，负责客户端TLS终结、请求/响应无改变转发、目的地固定集合解析；其他所有非敏感功能（认证、账户选择、计费、管理）留在不可信主机上，通过一个**不携带任何body字节的窄控制通道**与飞地通信。

- **关键技术细节**：
  1. **远程证明（Attestation）**：使用RA-TLS（Remote Attestation TLS）将飞地身份（测量值）、新鲜nonce、传输层公钥绑定在一条硬件签名的声明中。客户端在发送任何明文前必须先验证该声明。
  2. **测量锚定（Measurement Anchoring）**：通过公开的审计、可重现构建、透明日志（Audit, Reproduce, Pin, Attest – ARPA协议）确保客户端固定的是已知安全代码的哈希值。运营商发布构建清单并签名，存入append-only日志；客户端重现构建并检查日志一致性后锁定测量值。
  3. **最小可信基（Minimal TCB）**：飞地中仅包含851行忠实直通代码，不包含认证、路由选择、格式转换等逻辑，降低攻击面和审计难度。
  4. **形式化安全模型**：定义四个安全性游戏（保密性、目的地权威、忠实中继、流完整性），基于硬件安全假设和标准密码学原语进行Game-Based Reduction证明，将攻击者优势归约到底层假设（飞地隔离、执行完整性、证明不可伪造等）的突破概率。

- **算法流程（文字说明）**：
  - **引导阶段**：运营商`Publish`：构建代码→计算测量值→签名清单→写入透明日志。客户端`Audit, Pin`：验证签名、日志包含性和一致性→重现构建→锁定测量值m*。
  - **每会话阶段**：客户端`Vrfy`：接收飞地Quote→验证平台签名、测量值=m*、nonce匹配、绑定证书→仅成功后通过TLS释放请求体（加密）。
  - **每请求阶段**：飞地内`Relay`：解密请求体→调用主机控制通道获得目的地（仅名称）→在固化映射表`∆`中查找真实hostname→向该provider转发原样请求→将响应流加密后传回客户端→提取token用量报告给主机。

## 6. 论文的主要结论与发现

- **四种攻击全部被AEGIS阻断**：在明文基线下全部成功，在AEGIS保护下全部失败（因为主机无法读写明文，无法进行任何篡改或窃取）。
- **延迟极低**：稳态开销（飞地边界+加解密+虚机间通信）在常规小负载下仅为约**5.7 ms（中位数）**，2.0 ms增量（对比plain host relay 3.7 ms）。大payload（1 MB）时达52 ms，4 MB时183 ms。
- **对提供商透明**：支持三种原生API格式（OpenAI、OpenRouter、Gemini），无需转换层，100请求全部成功。
- **可审计**：两个编码代理能发现大部分植入违规（8/10和10/10），证明小规模忠实直通代码可由AI辅助审计。
- **形式化安全保证**：给出了基于归约的安全性证明，从数学上保证在硬件假设下AEGIS可实现机密性、完整性、目的地控制等目标。

## 7. 优点：方法或实验设计上的亮点

- **设计极简**：可信基仅851行代码，大大降低审计和形式化验证难度，且使测量值长期稳定（主机侧变更不影响）.
- **“发布前拒绝”而非“接收后验证”**：客户端在释放正文前强制完成证明验证，没有“事后检查”可能被跳过的窗口。
- **形式化证明完备**：定义四个游戏、三条定理、两条引理，全部给出具体优势界限，并使用Game-Based Shoup方法，逻辑严密。
- **实现可重现构建和透明日志**：运营商不能私下给不同客户端不同测量值，审计路径公开可信。
- **控制通道设计精巧**：仅传递账户标签、策略名称等元数据，**不传递任何地址或正文片度**，杜绝泄露风险。
- **实验设置分离了安全验证和性能测量**：安全测试使用真实攻击场景，性能测量使用本地上游隔离外部方差；流式响应也专门验证了分块时序没有额外抖动。

## 8. 不足与局限

- **残留侧信道**：飞地不实施流量整形，主机仍可通过密文记录大小和时序推知token长度序列，论文仅提出但这需要未来工作（如记录级填充）。
- **依赖平台信任**：假设AWS Nitro Enclaves的硬件根和证明机制安全，物理/微架构侧信道不在范围内；若存在突破（如CacheWarp、瞬态执行），安全性可能失效。
- **单平台实现**：仅评估了AWS Nitro，未测试Intel SGX、AMD SEV-SNP等其他TEE，可移植性未验证。
- **恶意提供商不在模型内**：论文假设提供商是诚实的；若提供商恶意发送篡改工具调用或窃取数据，AEGIS无法防御。
