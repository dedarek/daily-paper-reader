## 摘要
大型语言模型应用从模板构建提示，而Handlebars是一种广泛使用的模板引擎，也是Microsoft Semantic Kernel中默认的提示模板格式。其双花括号{x}表达式会对插值的值进行HTML转义，并作为安全默认值被文档记录；其三重花括号{x}表达式则以原始形式插入值。我们表明，这一选择悄无声息地决定了应用对结构角色注入的暴露程度，即攻击者控制的数据携带聊天角色分隔符，从而伪造出更高权限的轮次。一个无模型分析确立了其机制：Handlebars转义会重写尖括号，但不会重写方括号、冒号或Markdown井号，因此它能够中和ChatML、Llama-3和XML角色分隔符（存活率0.00），同时使Llama-2 [INST]、传统的Human:/Assistant:以及Markdown ###分隔符保持不变（后两者存活率1.00）。随后，我们在七个分隔符家族、两个攻击目标和四个模型（GPT-3.5 Turbo、GPT-4o mini、GPT-4.1 mini、Claude Haiku 4.5）上进行了5760次试验，总API成本为1.63美元。GPT-3.5 Turbo在97%的原始试验和91%的转义试验中遵循了任务劫持指令，其中转义保护集中在尖括号家族，而对基于冒号和Markdown的家族则没有保护；更难实现的秘密泄露目标（未饱和）更清晰地暴露了相同的家族交互。Claude Haiku 4.5几乎完全抵御了这两个目标。转义默认值仅保护那些HTML转义恰好覆盖其字符的分隔符方案，而对其余方案则不提供任何保护，并且无法替代指令与数据的结构分离。

## Abstract
Large language model applications build prompts from templates, and Handlebars is a widely used templating engine and the default prompt-template format in Microsoft Semantic Kernel. Its double-brace {x} expression HTML-escapes the interpolated value and is documented as the safe default; its triple-brace {x} expression inserts the value raw. We show that this choice silently governs an application's exposure to structural role injection, where attacker-controlled data carries chat role delimiters that forge a higher-privilege turn. A model-free analysis establishes the mechanism: Handlebars escaping rewrites angle brackets but not square brackets, colons, or Markdown hashes, so it neutralises ChatML, Llama-3, and XML role delimiters (survival rate 0.00) while leaving Llama-2 [INST], legacy Human:/Assistant:, and Markdown ### delimiters intact (survival rate 1.00 for the last two). We then run 5760 trials across seven delimiter families, two attack objectives, and four models (GPT-3.5 Turbo, GPT-4o mini, GPT-4.1 mini, Claude Haiku 4.5) at a combined API cost of 1.63 USD. GPT-3.5 Turbo follows the task-hijack instruction in 97% of raw and 91% of escaped trials, with the escaping protection concentrated in the angle-bracket families and absent for the colon- and Markdown-based families; the harder secret-exfiltration objective, which does not saturate, exposes the same family interaction more cleanly. Claude Haiku 4.5 resists both objectives almost entirely. The escaped default protects only the delimiter schemes whose characters HTML escaping happens to cover, gives no protection for the rest, and cannot substitute for a structural separation of instruction and data.

# 论文总结：Handlebars模板化LLM提示中的结构角色注入

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：大型语言模型（LLM）应用通常使用模板引擎动态构建提示（prompt）。Handlebars 是广泛使用的模板引擎，也是 Microsoft Semantic Kernel 的默认提示模板格式。它提供两种插值方式：双花括号 `{{ x }}`（默认，执行 HTML 转义，被文档标记为“安全”），以及三花括号 `{{{ x }}}`（原始插入，不转义）。然而，HTML 转义是为防御跨站脚本攻击（XSS）设计的，而 LLM 提示面临的是**结构角色注入**（Structural Role Injection）威胁：攻击者通过在受控数据字段中嵌入聊天角色分隔符（如 `<|im_start|>`、`[INST]`、`Human:` 等），伪造一个更高权限的轮次（如系统或助手角色），从而覆盖开发者的意图指令。

- **核心问题**：双花括号的 HTML 转义能否有效防御结构角色注入？其保护覆盖哪些角色分隔符？是否对所有分隔符家族都有效？当转义无效时，应用暴露的风险有多大？

- **整体含义**：本文系统揭示了 Handlebars 默认转义的保护范围完全由 HTML 字符集决定（仅覆盖尖括号），而对常见的方括号、冒号、Markdown 井号等分隔符家族完全无效。开发者若不意识到这一点，会误以为默认转义是安全的通用防御，实则留下严重的提示注入漏洞。该研究强调了**指令与数据的结构分离**远比依赖巧合的 HTML 转义更重要。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：通过**模型无关的静态分析**，独立于任何 LLM，直接计算各角色分隔符家族在 Handlebars 转义后的**字节级存活率**；再通过**端到端实证实验**，在真实模型上验证静态预测是否与实际攻击成功率（ASR）一致。两者结合，将防御效果的机制（转义函数）与模型行为分离。

- **关键技术细节**：
  - **静态分析**：对于每一家族的攻击载荷（break-out payload），使用 pybars3（Handlebars 的 Python 实现）的 `escapeExpression` 函数渲染。统计载荷中所有角色控制令牌（role-control tokens）在转义后有多少个仍以原始字节存在。存活率 = 幸存令牌种类数 / 总令牌种类数。
  - **Handlebars 转义字符集**：仅重写 `& < > " ' `=` 共 7 个字符，**不重写** `[` `]` `:` `#` 等。
  - **实证实验**：构建一个统一的实验管道（见图 2）：16 个场景 × 7 个分隔符家族 × 2 种攻击目标（任务劫持、秘密泄露）× 2 种插槽模式（原始/转义）× 4 个模型 × 3 次重复 = 5760 次试验。每个场景包含一个受信任的系统指令（内置唯一秘密金丝雀和禁止泄露指令）、合法任务、良性字段值以及注入指令。模板渲染后作为一个单条用户消息发送给 Chat API（系统消息仅为通用助手说明）。成功标准：任务劫持——模型输出攻击者指定的标记令牌；秘密泄露——模型输出系统指令中的秘密金丝雀。使用温度 0 以最小化随机性，但仍做 3 次重复以评估非确定性。统计方法：Wilson 置信区间、Fisher 精确检验、Cramér's V 效应量，零细胞时采用 Haldane–Anscombe 校正。

## 3. 实验设计

- **数据集/场景**：作者创建了 16 个不同应用类别的场景（文档摘要、客户支持、检索 QA、邮件分类、代码审查、内容审核、数据提取、翻译），每个类别两个场景。每个场景有独特的秘密金丝雀和合法任务。所有场景的良性字段值均不含注入内容。未使用公开基准数据集，而是自建场景，以控制变量并确保可重复性。

- **Benchmark**：没有明确的外部基准，但作者对比了四种模型（GPT-3.5 Turbo、GPT-4o mini、GPT-4.1 mini、Claude Haiku 4.5），覆盖从极易受攻击到几乎完全抵抗的范围。此外，每个模型都有一个**无注入的良性基线**（benign baseline），在所有试验中均产生零误报，确保任何成功反映真实注入。

- **对比方法**：主要对比变量是插槽模式（原始 `{{{ }}}` vs 转义 `{{ }}`）和分隔符家族（7种）。未与其他防御方法（如指令层次、StruQ、Spotlighting 等）直接对比，但论文在讨论中提及这些作为更佳的结构防御方案。

## 6. 主要结论与发现

1. **静态分析结果**（模型无关）：
   - ChatML、Llama-3、XML 家族（基于尖括号）的令牌存活率为 **0.00**（被完全转义）。
   - Markdown `###` 和 Anthropic `Human:/Assistant:` 家族存活率为 **1.00**（转义完全无效）。
   - Llama-2 家族存活率为 **0.33**（`<<SYS>>` 被转义，`[INST]` 幸存）。
   - 因此，默认转义的保护是**偶然的、不完整的**：仅覆盖 HTML 转义恰好处理的字符，对其他常见分隔符毫无作用。

2. **实证结果**（任务劫持）：
   - GPT-3.5 Turbo 在原始插槽中成功率 97%，转义后仍达 91%（差异较小，主要因为模型已饱和——即使无结构分隔符的 `plain` 家族成功率也达 96%）。
   - GPT-4o mini 展示了最清晰的保护效果：`llama3` 家族从 56% 降到 12%，`xml` 从 94% 降到 75%，`llama2` 从 33% 降到 0%，而 `anthropic` 和 `markdown` 无变化。
   - GPT-4.1 mini 几乎和 GPT-3.5 Turbo 一样易受攻击（96% 原始，94% 转义），新模型并不更安全。
   - Claude Haiku 4.5 几乎完全抵抗（0% 成功率），转义与否无影响。

3. **实证结果**（秘密泄露）：
   - 整体成功率远低于任务劫持，因为模型更难泄露受保护秘密。GPT-4.1 mini 最高（35% 原始，29% 转义），GPT-4o mini 为 13%，Claude Haiku 4.5 为 0%。
   - 家族交互关系更清晰：GPT-3.5 Turbo 中，`llama3` 家族从 50% 降到 35%，`xml` 从 35% 降到 27%，而 `markdown` 和 `anthropic` 不变。

4. **最终结论**：
   - 默认转义**不能替代指令与数据的结构分离**。它是免费且无害的（应保持开启），但必须与真正的结构防御（如指令层次、结构化查询、数据标记）配合。
   - 开发者在选择提示模板格式时，应明确自己使用的角色分隔符家族，并对未受保护的字符（方括号、冒号、井号）采取额外中和措施。
   - 模型本身的抵抗力是决定风险的最大因素（比插槽模式影响更大），且模型“年龄”不能预测安全性。

## 7. 优点

- **方法论创新**：将模型无关的静态分析与实证实验相结合，清晰隔离了转义机制和模型行为，使结论具有一般性，不依赖于特定模型。
- **问题界定清晰**：直接戳穿“HTML 转义即可防御提示注入”的常见误解，给出了具体量化证据。
- **成本低廉**：仅花费 1.63 美元 API 费用完成 5760 次试验，实验设计高效。
- **可重复性**：作者承诺开源所有脚本、场景定义、分析代码，并提供独立的验证脚本 `verify_numbers.py`，结果可逐行追溯。
- **多模型并不同世代覆盖**：包括 GPT-3.5、GPT-4 系列、Claude Haiku，证明了不同模型安全性的巨大差异，也为研究者提供了横向比较基准。
- **客观稳健的统计**：使用 Wilson 区间、Fisher 精确检验，报告效应量，且对零细胞做了保守校正。3 次重复（温度 0）验证了结果稳定。

## 8. 不足与局限

- **场景数量有限**：仅 16 个场景，导致每家族每条件样本量较小（3 次重复后 48 个），个体家族差异的置信区间宽泛，作者也承认“individual family differences are not individually significant”。
- **仅测试温度 0**：虽然温度 0 减少随机性，但实际部署中可能使用非零温度，导致结果可能略有不同。不过温度 0 更利于复现和因果推断。
- **API 单条消息场景**：论文将扁平化模板作为单条用户消息发送，未测试本地模型栈（如 llama.cpp、vLLM）中对特殊令牌的原生解析。作者指出本地栈可能放大安全风险（因为幸存的分隔符会触发真正的角色切换），因此本文结果应视为“保守下界”。
- **仅一个模板引擎**：Handlebars 的转义规则与其他模板引擎（如 Jinja2 的 `autoescape`、Mustache）不同，结论不能直接迁移。需要单独分析。
- **未对比其他防御方法**：虽然讨论了指令层次、StruQ、Spotlighting 等，但未在实验中将其作为对照组，因此无法直接量化 Handlebars 转义与这些防护的相对收益。
- **攻击仅涉及文本角色分隔符**：未考虑其他形式的注入（如多模态内容、代码块内的指令等）。也未测试多轮对话场景，仅测试单轮注入。
- **成功标准是严格的且可能漏报**：任务劫持需要模型输出特定标记令牌，秘密泄露需要输出金丝雀字符串。模型可能部分遵守但未输出精确目标（如泄露了部分秘密但未完整输出金丝雀），导致假阴性。然而良性基线零误报表明这些标准是保守的。
- **商业 API 黑箱**：调用的是闭源模型，无法获得模型内部状态（如 logits、注意力），因此无法进一步分析为什么某些模型对某些分隔符更敏感。
